**A team of agents and the model they run on, in one Node process.**

The HDK is an in-app intelligence runtime, not an orchestration framework. There's no model server and no text seam — the model's KV cache and your application's state share one process, and agents are branches of the model's live attention. `npm i`, point at a GGUF on disk, and this runs:

```typescript
import { main, call } from "effection";
import { createContext } from "@lloyal-labs/lloyal.node";
import { initAgents, agentPool, parallel } from "@lloyal-labs/lloyal-agents";
import { reportTool } from "@lloyal-labs/rig";

main(function* () {
  const ctx = yield* call(() =>
    createContext({ modelPath: "model.gguf", nCtx: 32768, nSeqMax: 8 }),
  );
  yield* initAgents(ctx);

  const pool = yield* agentPool({
    orchestrate: parallel(
      questions.map((q) => ({ content: q, systemPrompt: RESEARCH_PROMPT })),
    ),
    tools: [searchTool, reportTool],
    terminal: reportTool,
  });
  // pool.agents[0].result, pool.agents[1].result, ...
});
```

Three agents just ran concurrently on hardware that serves one. They shared a system prompt and tool schemas **physically** — decoded once into KV cache, inherited by every fork — not re-sent as strings on every step. Frameworks like LangChainJS pass *application state* between stateless model calls: variables, conclusions, scratchpads, while the model re-reads everything from scratch. The HDK shares *model state* — the attention over what has already been read.

That difference is why this ships inside an Electron app, an `npm install -g` CLI, or a worker in an existing Node backend — and why every agent tick runs on the user's hardware at zero marginal cost to you. What you build splits in two: a **harness** is the orchestrated system of agents you ship — your UI, your pipeline, your persistence. An **AgentApp** is a packaged capability — a `Source` plus `Tool`s plus a prompted skill — that a harness loads at runtime. Same author can wear both hats on day one.

## Continuous Context

Most agent frameworks pass strings between stages: a transcript of what one agent generated becomes a prompt for the next, and the model's prior attention state is discarded and rebuilt each call. lloyal agents fork instead. A fork is metadata-only — O(1), zero tensor copy — and when stage 2 starts decoding, stage 1's findings are already in its KV at their original positions; its first generated token attends over them directly. ([Continuous Context](/under-the-hood/continuous-context-spine) has the physics.)

At its core, **an agent is a branch with intent** — a forked KV slot, a system prompt, a tool set, a terminal condition. Pools are sets of agents on a shared spine; harnesses orchestrate pools. The whole stack is branches all the way down.

This is what breaks the "runs locally but is a toy" barrier. Re-encoding ~900 tokens of system prompt and tool schemas per agent step is prohibitive on consumer hardware — most local-LLM agent demos fall off a performance cliff at the second hop. With Continuous Context the prefix is decoded once and inherited by every fork: measured across a real pipeline, **4.4× fewer tokens processed** than prompt rebuilding. The same harness code runs on a 32K cloud GPU or a 16K laptop — depth adapts to the hardware.

## Deep agents, native

Elsewhere, a sub-agent is a fresh conversation handed a task brief — everything the parent knew gets squeezed through a prompt and re-paid for. The pattern is valuable enough that LangChain productized it separately as DeepAgents. Here it isn't a product; it falls out of the architecture. A sub-agent is a fork of the parent's live attention: it inherits every search result, page read, and reasoning step — instantly, for the cost of a turn separator. Or it forks from the clean shared spine when you *want* quarantine. Isolation is a fork-point choice, not a mandate.

Recursion is structural, at two levels:

- **Model level** — a tool's `execute()` returns an Effection `Operation`, so it can `yield*` straight into `agentPool`. An agent that calls such a tool spawns sub-agents mid-generation, inside its own scope. `DelegateTool` is the canonical implementation.
- **Harness level** — a completed pool's branches stay alive; follow-up pools fork from any agent's frontier.

And because delegation happens against live attention, the boundary can be *judged*: `DelegateTool` scores proposed sub-tasks against the original query (entailment gating) and against the delegating agent's own task (echo detection — "your sub-questions paraphrase your own task; report instead of re-delegating"). An orchestration framework can't run that check; it has no attention state to score against.

Depth is unbounded — sub-agents can delegate further — and every subtree is a structured-concurrency scope: halt it and its KV is reclaimed, guaranteed. The deeper an investigation goes, the richer the attention state at depth. **Depth adds context, not cost.**

## Retrieval Interleaved Generation

Classical RAG retrieves before the model thinks: embed the query, fetch top-K by similarity, paste into the prompt, generate once. RIG retrieves *while* it thinks. The agent emits a tool call → the AgentApp's `Source` returns chunks → a cross-encoder reranker scores them against the original intent → results prefill into the agent's KV at the call position → the next decoded token attends over them and decides what to ask for next.

The difference is observable in tool-call inputs. A RAG system constructs queries from the user's question; a RIG agent constructs queries from hypotheses formed during generation:

```
grep(/memory leak/)            → 3 matches in 2 files
read_file(pool.ts L40-80)      → reads allocation logic, spots missing cleanup
search("resource cleanup on connection close")  → finds teardown handler
read_file(server.ts L120-155)  → discovers close handler never calls pool.drain()
grep(/drain|dispose|cleanup/)  → 8 matches, confirms drain exists but is unused
search("pool drain connection lifecycle interaction")  → targets the gap
report(findings)
```

The last search is the signature behavior: the agent discovered `drain()` existed but was never called on connection close, and constructed a query targeting exactly that interaction. Multi-hop reasoning, emergent from attention mechanics — not a prompted retry loop.

Behind this sit the pieces the kit ships: two reference AgentApps (**`web_research`**, **`corpus_research`**) to read as worked examples; the reranker's scoring roles and pressure-aware explore/exploit ([Sources & retrieval](/build-an-app/sources-and-retrieval)); per-spawn prompt templating; and `protected` tools gated by `authGuard` at dispatch ([AgentApp security](/build-an-app/app-security)).

## The honest comparison

Full stack against full stack:

| Capability         | lloyal HDK                              | Typical agentic stack                            |
| ------------------ | --------------------------------------- | ------------------------------------------------ |
| Inference          | embedded native binding                 | ollama / LM Studio + HTTP                        |
| Agent loop         | built-in                                | langchain / langgraph                            |
| Tool dispatch      | `Tool<TArgs>` interface (+ `protected`) | langchain Tools + glue                           |
| Retrieval          | `Source` + native cross-encoder reranker | langchain RAG + vector DB + embedder            |
| Multi-agent shapes | `parallel` / `chain` / `fanout` / `dag` | langgraph / autogen / crewai                     |
| Multi-turn state   | `Session.trunk` (KV-backed)             | conversation-memory class + persistence          |
| Capability packaging | **AgentApps** (`defineApp`: Source + Tools + skill) | per-server MCP / bespoke integrations    |
| Process model      | **one Node process**                    | **inference server + agent process + DB**        |
| Distribution unit  | **npm package, ships with your app**    | **Docker compose / k8s for the back-end pieces** |

The asymmetry is the point. Each row of the right column is a service to install, configure, version, secure, and orchestrate — and an HTTP boundary your agent state gets serialized across. Each row of the left column is an import. (Dependency footprint: `@lloyal-labs/sdk` has zero production dependencies; `lloyal-agents` adds [`effection`](https://frontside.com/effection/) and [`eta`](https://eta.js.org/); `rig` adds `@mozilla/readability` + `linkedom`.)

## The stack

```
Your application
    ↓
Apps (defineApp + manifest)         ← what third parties build (corpus, web, custom)
    ↓
@lloyal-labs/rig                     Registry, retrieval primitives, harness building blocks
    ↓
@lloyal-labs/lloyal-agents           Continuous Context agent orchestration, authGuard
    ↓
@lloyal-labs/sdk                     Branches, sessions, KV cache, reranker
    ↓
@lloyal-labs/lloyal.node             Node-native binding (JS ↔ C++)
    ↓
liblloyal                            C++ core — KV machinery, sampler, GPU integration
                                     ─ goes wherever C++ runs ─
```

Four npm packages over a portable C++ core. [**liblloyal**](https://lloyal-ai.github.io/liblloyal/) runs anywhere a C++ toolchain and a compute backend exist — desktop, mobile, WASM, game engines; the Node binding is the surface that ships today, with prebuilt binaries for macOS (Metal), Linux and Windows (CPU / CUDA / Vulkan).

## Hardware floor

| Requirement      | Floor for a working harness                                                                                                                              |
| ---------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------- |
| RAM              | ~3 GB resident for a 4B-class model at Q4_K_M, plus ~600 MB for a small reranker, plus KV cache. 8 GB total system RAM is comfortable.                   |
| Compute backend  | GPU recommended (Metal / CUDA / Vulkan / ROCm). CPU works for tiny models or as fallback. Pure MCU-class embedded (no accelerator, no spare RAM) is out. |
| Storage          | ~3 GB for weights of a 4B-class model at Q4; varies with model size.                                                                                     |
| Binary footprint | ~150–300 MB of compiled liblloyal + binding code, depending on which compute backends are statically linked.                                             |

The production floor reference harnesses target — three concurrent agents, web + corpus retrieval, 32K context, on a 16 GB consumer laptop:

- **Weights**: any 4B-class instruction-tuned GGUF llama.cpp loads — **Qwen 3.5** is the recommended default — at **`Q4_K_M`** (~3 GB resident). The model is a configuration choice, not a constraint.
- **KV cache**: `typeK: "q4_0"`, `typeV: "q4_0"` — FP16 KV would 4× the cache footprint; Q4 KV is the difference between fitting a 32K window on consumer hardware and not.
- **Reranker**: a 0.5–0.6B cross-encoder GGUF at Q8_0 (~600 MB resident).
- **Bootstrap**: `nCtx: 32768` on a single `llama_context`.

Quality holds at this floor, throughput is interactive, and the whole stack — model, reranker, KV, your code — fits in well under 8 GB resident. SSM and recurrent models (Mamba, RWKV) are first-class via llama.cpp; their fixed-size state composes through fork-and-extend even better than a linearly growing KV cache.
