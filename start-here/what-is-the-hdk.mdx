---
title: What is a harness?
description: The model is maybe a fifth of the product. The harness is the other four-fifths — and it should hold the model, not call it across a wire.
---

A harness is everything built around a model to turn it into working software: the agent loop, the tools, the context management, the prompts and policies, the surface that shows a person what's happening. The model provides raw capability; the harness makes it do a job. Agentic coding tools are harnesses around frontier models. Eval rigs are harnesses. Artifact, our deep-research app, is a harness. The industry converged on the word because every serious team discovers the same thing: the model is maybe a fifth of the product, and the other four-fifths needed a name.

If you've shipped anything real on a model, you've built one — whether or not you called it that.

So this page could end here. That's the definition, it's the standard one, and it's the one we use. But before you move on, there's a question worth sitting with — the one this entire kit is built on. Take the harness you've built, or the one you're planning, and ask:

## Where's the model?

Not which model — *where*. Sketch your architecture and put your finger on it.

For nearly every harness ever shipped, your finger just landed on the far side of an HTTP boundary. The harness lives in one process, the model in another, and between them runs a completions interface: everything your harness will ever know about its model arrives as text through that slot. This holds even when the model sits on your own machine. A llama-server on localhost is a hosted API with a three-millisecond commute — the wire got shorter; the architecture never moved.

Nobody chose this. Models first became useful behind hosted endpoints, so the whole craft of harness-building grew up on the far side of that seam — and the seam, quietly, decided what a harness is allowed to be. It has been there so long it reads as physics. It isn't. It's an architecture, and architectures decide things. This one has made four decisions on your behalf:

**Your agents cannot share a working memory.** The most valuable object in the system — the attention state built up over a context — lives below the seam, and your harness can only manage it by remote control, through text. So every agent is a stranger to every other agent: shared context gets re-sent and re-computed, turn after turn. (The cost shadow of this isolation is measurable — seam-style orchestration processes 4.4× more tokens than the same pipeline run below it — but the cost is the symptom. The isolation is the condition.)

**You cannot see or steer below the string.** Down there are logits, entropy, grammar state, sampler chains — the signals that would tell you your model is uncertain, the levers that would let you constrain it token by token. The completions interface launders all of it into prose. You're flying the plane through a mail slot.

**You cannot answer "why."** Someone asks why the agent did that. What have you got? Request logs from four programs with four clocks. Causal structure dies at a network boundary; you can't replay what was never one process.

**Your agents don't exist between requests.** Try to point at one — not its prompt template, the running thing. You can't, because it isn't anywhere: its state is smeared across a client, a server, and a transcript. It can't be paused, moved, or resumed, because there is no *it*.

None of this is a defect in your framework. These are properties of the seam itself — which is why switching frameworks has never once given any of it back.

## So strip the ceremony

Put the model *inside* the harness. Model, agents, tools, policy — one process, and your orchestration operates on attention state directly instead of on text across a boundary. That's the entire move.

Here it is as code. `npm i`, point at a GGUF on disk, and this runs:

```typescript
import { main, call } from "effection";
import { createContext } from "@lloyal-labs/lloyal.node";
import { initAgents, agentPool, parallel } from "@lloyal-labs/lloyal-agents";
import { reportTool } from "@lloyal-labs/rig";

main(function* () {
  const ctx = yield* call(() =>
    createContext({ modelPath: "model.gguf", nCtx: 32768, nSeqMax: 8 }),
  );
  yield* initAgents(ctx);

  const pool = yield* agentPool({
    orchestrate: parallel(
      questions.map((q) => ({ content: q, systemPrompt: RESEARCH_PROMPT })),
    ),
    tools: [searchTool, reportTool],
    terminal: reportTool,
  });
  // pool.agents[0].result, pool.agents[1].result, ...
});
```

Three agents just ran concurrently on hardware that serves one. They shared a system prompt and tool schemas **physically** — decoded once into KV, inherited by every fork — not re-sent as strings each step. The same code ships inside an Electron app, an `npm install -g` CLI, or a worker in an existing Node backend, and every tick runs on the user's hardware at zero marginal cost to you.

Everything else falls out of the one move — and each thing that falls out is something you've probably wanted and assumed you couldn't have:

**Agents become cheap enough to spend.** In-process, spawning an agent is a copy-on-write fork over shared KV — a metadata write, not a re-computation. A whole cohort decodes at one GPU dispatch per tick regardless of how many agents you're running; you pay for concurrency in memory cells, not compute. So go ahead: fan out six hypotheses. Probe your sources in parallel. Run a landscape survey and let every later agent inherit its findings for free, because they fork from the spine it extended. On a laptop.

**Context becomes a governed resource.** With the KV addressable, your harness gets to do what an operating system does for memory: admission control, pressure floors, recovery reserves. Agents spawn only when there's room to feed them, get nudged to wrap up as pressure rises, and are reaped — findings salvaged, not lost — before the cache can overflow. Structured concurrency extends the guarantee to the GPU itself: when a scope closes, its branches are pruned and their state is reclaimed. You will never leak attention.

**Reasoning moves inside the token loop.** Your model can stop mid-thought, retrieve, and continue — retrieval and entailment scoring interleaved with decoding itself, not staged between prompts. We call it retrieval-interleaved generation, and it's only expressible when the token loop is yours. It shows up in the tool-call inputs: a RIG agent constructs queries from hypotheses formed *during* generation, not from the user's original question.

```
grep(/memory leak/)            → 3 matches in 2 files
read_file(pool.ts L40-80)      → reads allocation logic, spots missing cleanup
search("resource cleanup on connection close")  → finds teardown handler
read_file(server.ts L120-155)  → close handler never calls pool.drain()
grep(/drain|dispose|cleanup/)  → drain exists but is unused
search("pool drain connection lifecycle interaction")  → targets the gap
report(findings)
```

That last search is the signature move: the agent discovered `drain()` existed but was never called on connection close, and built a query aimed at exactly that interaction. Multi-hop reasoning, emergent from attention mechanics — not a prompted retry loop.

**Your agents get an address.** In-process, an agent's state is fully described by tokens plus fork topology plus seeds — all recorded. Which means you can suspend one, serialize it to a few kilobytes of *readable text*, and resume it somewhere else. Another process. Another machine. Another GPU backend. Your user starts a research run on their phone and finishes it on their workstation, and it's the same agent — not a new one that read the old one's diary.

**Your exhaust becomes an asset.** One process, one clock, one causal record: every fork, tool call, retrieval decision, and outcome — replayable, deterministically, under pinned settings. That's your audit log and your debugger. It's also something more: forked siblings share exact state *by construction*, which makes your traces training data in precisely the shape post-training methods want. The harness improves the model that runs it. Keep your exhaust.

**Your transparency stops being decorative.** Capabilities are declared entitlements, enforced at the tool gate, and emitted as runtime events your UI can only relay — never author. When your harness shows the user that an app touched the network or read their files, that indicator was generated *below* any code that could have lied about it. Your privacy story becomes a property you can prove, not a promise you have to make.

## What we're not telling you

We're not telling you the seam-built harness is wrong for every job. If you're serving a thousand concurrent chat users, that's a batching and capacity problem, the platforms built for it are genuinely good at it, and an in-process harness — single-tenant by design, because determinism and multi-tenant batching cannot coexist — shouldn't pretend otherwise. A harness serves *sessions and outcomes*: one user's research run, one team's pipeline, one workflow's agents finishing a job.

And we're not telling you to give up deployment flexibility. "The model lives inside the harness" is a claim about software topology, not about where the silicon sits. The same harness runs on a phone, a laptop, a perimeter server, or in a VPC. When something does cross a machine boundary, it's tasks in and telemetry out. Your agent turns never touch a wire.

## What "development kit" means

Everything above is built and shipping. HDK — the Harness Development Kit — is that toolchain: the C++ inference kernel and its Node bindings (on npm today, prebuilt for thirteen platform and GPU targets), the structured-concurrency runtime with its reaping guarantees, the agent pool and orchestrators, the fork mechanics and pressure governance, the replayable trace format, and the signed AgentApps channel through which your harness gains new capabilities without gaining unreviewed code. You write the harness; the kit supplies the machinery a harness deserves. Four npm packages over a portable C++ core:

```
Your application
    ↓
AgentApps (defineApp + manifest)     ← what third parties build (corpus, web, custom)
    ↓
@lloyal-labs/rig                     Registry, retrieval primitives, harness building blocks
    ↓
@lloyal-labs/lloyal-agents           Continuous Context agent orchestration, authGuard
    ↓
@lloyal-labs/sdk                     Branches, sessions, KV cache, reranker
    ↓
@lloyal-labs/lloyal.node             Node-native binding (JS ↔ C++)
    ↓
liblloyal                            C++ core — KV machinery, sampler, GPU integration
                                     ─ goes wherever C++ runs ─
```

[**liblloyal**](https://lloyal-ai.github.io/liblloyal/) runs anywhere a C++ toolchain and a compute backend exist — desktop, mobile, WASM, game engines; the Node binding is the surface that ships today, with prebuilt binaries for macOS (Metal), Linux and Windows (CPU / CUDA / Vulkan).

### The honest comparison

Full stack against full stack:

| Capability         | lloyal HDK                              | Typical agentic stack                            |
| ------------------ | --------------------------------------- | ------------------------------------------------ |
| Inference          | embedded native binding                 | ollama / LM Studio + HTTP                        |
| Agent loop         | built-in                                | langchain / langgraph                            |
| Tool dispatch      | `Tool<TArgs>` interface (+ `protected`) | langchain Tools + glue                           |
| Retrieval          | `Source` + native cross-encoder reranker | langchain RAG + vector DB + embedder            |
| Multi-agent shapes | `parallel` / `chain` / `fanout` / `dag` | langgraph / autogen / crewai                     |
| Multi-turn state   | `Session.trunk` (KV-backed)             | conversation-memory class + persistence          |
| Capability packaging | **AgentApps** (`defineApp`: Source + Tools + skill) | per-server MCP / bespoke integrations    |
| Process model      | **one Node process**                    | **inference server + agent process + DB**        |
| Distribution unit  | **npm package, ships with your app**    | **Docker compose / k8s for the back-end pieces** |

The asymmetry is the point. Each row of the right column is a service to install, configure, version, secure, and orchestrate — and an HTTP boundary your agent state gets serialized across. Each row of the left column is an import. (`@lloyal-labs/sdk` has zero production dependencies; `lloyal-agents` adds [`effection`](https://frontside.com/effection/) and [`eta`](https://eta.js.org/); `rig` adds `@mozilla/readability` + `linkedom`.)

### What it takes to run

The production floor the reference harnesses target — three concurrent agents, web + corpus retrieval, 32K context, on a 16 GB consumer laptop:

- **Weights**: any 4B-class instruction-tuned GGUF llama.cpp loads — **Qwen 3.5** is the recommended default — at **`Q4_K_M`** (~3 GB resident). The model is a configuration choice, not a constraint.
- **KV cache**: `typeK: "q4_0"`, `typeV: "q4_0"` — FP16 KV would 4× the footprint; Q4 KV is the difference between fitting a 32K window on consumer hardware and not.
- **Reranker**: a 0.5–0.6B cross-encoder GGUF at Q8_0 (~600 MB resident).
- **Bootstrap**: `nCtx: 32768` on a single `llama_context`.

Quality holds at this floor, throughput is interactive, and the whole stack — model, reranker, KV, your code — fits in well under 8 GB resident. SSM and recurrent models (Mamba, RWKV) are first-class via llama.cpp; their fixed-size state composes through fork-and-extend even better than a linearly growing KV cache.

Artifact is the existence proof — multi-agent, source-grounded research with a live transparency surface, running entirely on a consumer machine, built with the same kit you're about to use.
