description: On-device inference, offline operation, dispatch-gated writes, signed disclosure, and a provable audit trail — private AI as an architectural property, not a policy promise.

**Your agent turns never touch a wire.**

The model runs inside your application, on the user's machine. There is no inference API, no completions endpoint, no prompt or token crossing a network boundary to be processed. That single architectural fact — [the harness holds the model](/start-here/what-is-the-hdk) — is what makes every claim on this page a *property you can verify* rather than a policy you have to trust.

This page is the whole egress story for a harness built on the HDK: what leaves the machine, when, under whose control, and how you prove it.

## What leaves the machine

Four categories cover every byte:

**1 · Inference: nothing.** Prompts, documents, the model's reasoning, tool results, the KV cache — all of it lives and dies in one local process. A research run over your files is computed entirely on your silicon. There is no per-token bill because there is no per-token wire.

**2 · The framework itself: nothing.** The runtime packages (`@lloyal-labs/sdk`, `lloyal-agents`, `rig`) make **no network calls of their own** — no telemetry, no analytics, no phone-home, no update checks. (`@lloyal-labs/sdk` has zero production dependencies; the others add only local-parsing libraries.) The only network code in the framework is the signed-channel *verifier*, and it runs at install time, not at runtime.

**3 · Tools: only what an enabled AgentApp declares — and you can see every call.** Network access exists exclusively inside the tools of AgentApps the harness operator chose to enable. A `web_search` tool reaches a search API; a `wikipedia` tool reaches Wikipedia's REST API; a corpus tool reaches nothing. Every invocation is emitted as a runtime event (`agent:tool_call` / `agent:tool_result`) the moment it happens.

**4 · Install time: one signed fetch, on your terms.** `harness.dev install` fetches a signed tarball from the catalog once; after that, the dependency reproduces from the lockfile. Nothing about a run — queries, documents, results — is ever sent back.

## Offline and air-gapped

Once the model weights and npm packages are on disk, inference needs no network at all. A harness whose AgentApps are local-only (the `corpus` reference app: your files, indexed in-process, retrieved in-process) runs **fully offline** — laptop in airplane mode, workstation on an isolated VLAN, an air-gapped environment where the packages and weights arrive on physical media. Tools that need the network simply aren't enabled; nothing degrades silently, because nothing else ever wanted the network in the first place.

Your documents never leave: corpus indexing, chunking, reranking, and retrieval all happen inside the process. Data residency is trivially satisfied — the data never has a chance to move.

## Enforcement, not promises

Consequential actions are gated **at dispatch**, below any code that could misrepresent them:

- A tool that writes, sends, or exfiltrates is marked **`protected`**. The framework's **authGuard** denies every protected call unless the session holds an explicit **grant** for it — fail-closed: no grant store configured means every protected tool is denied. Credentials never enter the model's context; the model can only *trigger* a gated call, never see or replay the secret.
- The gate runs in the agent runtime, not in the UI. When a harness shows the user "this app sent an email," that indicator was generated *below* any application code that could have lied about it — the UI can relay runtime events, never author them.

[AgentApp security](/build-an-app/app-security) covers the full threat model, including prompt injection.

## Disclosure you can read before anything runs

Every AgentApp on the signed channel ships a signed **attention surface** — the complete skill prose, every tool schema, and its `useWhen` routing sentence — generated at publish, reviewed before signing, and Ed25519-verified at install. The catalog entry also carries the app's **entitlements**: the human-readable disclosure of what the app touches (network, files, accounts), reviewed by the channel and rendered in the storefront. You know what an app can do before it is ever loaded, and the bytes you install are exactly the bytes that were reviewed — [the install verifies signatures end-to-end](/cli/installing) and audits the lockfile afterward.

## The audit trail is the architecture

One process means one clock and one causal record. Every session can write a complete trace — every fork, prompt, tool call, retrieval decision, and outcome — as local JSONL, **replayable deterministically** under pinned settings. "Why did the agent do that?" has an answer you can re-execute, not a correlation across four services' logs. See [Tracing](/build-a-harness/tracing).

The trace is yours, on your disk. Exporting any of it is a choice your harness makes explicitly — the framework never does it for you.

## Deployment topology

"The model lives inside the harness" is a claim about software topology, not about where the silicon sits. The same harness runs on a phone, a laptop, a perimeter server, or in a VPC. When something does legitimately cross a machine boundary in your architecture, it's **tasks in and results out** — your protocol, your choice of payload. The agent turns themselves — prompts, attention state, reasoning — stay on the machine doing the thinking.

## What this page does not claim

Honesty is part of the property. An enabled AgentApp's *open* (non-`protected`) tools execute their own code, and if that code reaches the network, it can — that's what web research *is*. The controls are exactly the ones above: you choose which apps to enable, the signed channel reviews and discloses what each one does, every call is visible as it happens, and anything consequential sits behind a grant. Zero egress is the **default state** of a harness — and every departure from it is declared, gated, and observable.
