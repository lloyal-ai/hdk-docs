**An agent just vanished from your pool. This page tells you why — and which knob fixes it.**

Every drop is a `pool:agentDrop` trace event with a reason. Match what you're seeing, jump to the enforcement point:

| You're seeing | Drop reason | First fix |
|---|---|---|
| Fewer agents ran than you spawned | `pressure_init` | smaller per-agent suffix, or raise `nCtx` |
| Agent killed mid-investigation, findings still landed | `pressure_critical` | working as designed — recovery extracted in the `hardLimit` reserve; raise `softLimit` to nudge earlier |
| `Invariant Violation — hardLimit must be >= nBatch` at startup | — | `budget: { context: { hardLimit: 1024 } }` |
| Tool results never settle, pool stalls, then drops | `pressure_settle_reject` / `settle_stall_break` | smaller tool payloads; `pruneOnReturn: true` |
| Agents nudged "report within N words" | (no drop) | over the soft budget — normal backpressure |
| Agent ended without result and without a drop | `maxTurns` / `free_text_stop` | raise `maxTurns`; configure `recovery` so the post-loop pass extracts |

This page documents `agent-pool.ts` (the pool's enforcement mechanics) and `AgentPolicy.ts` (the judgment layer) in `@lloyal-labs/lloyal-agents`. The pool handles mechanics — transitions, prefills, prunes. The policy handles judgment. For per-deployment tuning of `nCtx` / limits / KV quantization, see [Deployment](/guides/deployment).

## The budget model

Agent pools share a single KV cache. `ContextPressure` reads `SessionContext._storeKvPressure()` — `{ nCtx, cellsUsed, remaining }` where `remaining = nCtx - cellsUsed` — and two thresholds partition `remaining` into three zones:

```
                nCtx
 ┌──────────┬───────────────────┬──────────┬────────┐
 │cellsUsed │    headroom > 0   │ softLimit│hardLim │
 │ (in use) │   (new work OK)   │(reserved)│(recov) │
 └──────────┴───────────────────┴──────────┴────────┘
              <- remaining ->
                                |          |
 headroom = remaining - softLimit          |
 critical = remaining < hardLimit ---------┘
```

- **headroom > 0** — room for new work. Tool results can be prefilled, agents can continue generating.
- **headroom ≤ 0** — over budget. SETTLE defers tool results. PRODUCE nudges the model to wrap up.
- **critical** — `remaining` has dropped below `hardLimit`. Agents are killed before `produceSync()` to prevent `llama_decode` crashes. The `hardLimit` reserve is the recovery extraction budget.

Pressure is *per-context* (one `llama_context`, one `nCtx` budget), not per-AgentApp — every enabled AgentApp's tools draw from the same budget once their results prefill into agent branches. AgentApp lifecycle teardown is a separate system: the registry's per-AgentApp scopes free *external* allocations (corpus chunks, watchers) on scope halt; pressure frees *KV cells* on the tick-loop's schedule.

## The `hardLimit >= nBatch` invariant

`hardLimit` isn't an arbitrary safety margin — it's the **parallel report budget** for the recovery path. When `pressure.critical` fires and the kill proceeds, `recoverInline` prefills its extraction prompt and runs a single-agent produce/commit loop *inside* the reserved `hardLimit` cells. If that reserve is smaller than the context's `nBatch`, the first recovery decode fails with `llama_decode: failed to find a memory slot for batch of size N` — native OOM.

The pool enforces this at `useAgentPool` entry. If `hardLimit < nBatch`, the pool throws:

```
useAgentPool: Invariant Violation — hardLimit (128) must be >= nBatch (512).
Recovery reserves hardLimit cells for its own decode; if smaller than nBatch,
the next batch allocation will OOM. Increase policy.budget.context.hardLimit
to at least 512.
```

Misconfigured policies fail loudly before any agents spawn.

<Note>
Until `SessionContext.nBatch` is exposed natively, the pool validates against `ContextPressure.ASSUMED_N_BATCH = 512` (llama.cpp's default `n_batch`). Once the native exposure lands, the check reads `ctx.nBatch` directly so policies running against a custom batch size validate against the actual value.
</Note>

## `ContextPressure` API

Constructed from a `SessionContext` and thresholds from the policy. Each instance is an **immutable snapshot**, frozen at phase boundaries so all decisions within a phase evaluate against the same baseline — without this, items processed earlier in a loop would see different pressure than items processed later, making reject/nudge/kill decisions order-dependent.

```typescript
const policy = new DefaultAgentPolicy({
  budget: { context: { softLimit: 1024, hardLimit: 1024 } },
});

pressure.nCtx              // total KV cache capacity (0 when no limit)
pressure.cellsUsed         // KV cells currently in use
pressure.remaining         // nCtx - cellsUsed (Infinity when nCtx <= 0)
pressure.headroom          // remaining - softLimit
pressure.critical          // remaining < hardLimit
pressure.canFit(n)         // n <= headroom
pressure.percentAvailable  // 0–100 integer: remaining / nCtx * 100
```

`percentAvailable` is the single source of truth for KV availability: it drives `policy.shouldExplore()`, lands on `ToolHistoryEntry.contextAfterPercent` after each settle, and is injected into tool results as `_contextAvailablePercent` so agents can make pressure-aware decisions:

```json
{"matches": [...], "totalMatches": 4, "_contextAvailablePercent": 32}
```

**Defaults:** `softLimit = 1024`, `hardLimit = 512` (matches llama.cpp's default `n_batch`). Pools that use recovery with a larger expected report body should raise `hardLimit` — e.g. `1024` to fit a ~400-token extraction.

## The policy interface

Every lifecycle decision passes through an `AgentPolicy`:

```typescript
interface AgentPolicy {
  onProduced(agent, parsed, pressure, config): ProduceAction;
  onSettleReject(agent, resultTokens, pressure, config): SettleAction;
  shouldExplore?(agent, pressure): boolean;
  shouldExit?(agent, pressure): boolean;
  onRecovery?(agent, pressure): RecoveryAction;
  onToolRetry?(agent, tool, error, attempt): ToolRetryAction;
  readonly pressureThresholds?: PressureThresholds;
  resetTick?(): void;
}
```

| Hook | Phase | When | What it decides |
|------|-------|------|-----------------|
| `shouldExit` | PRODUCE (pre-generate) | Before each agent produces tokens | Kill immediately — no chance to respond |
| `onProduced` | PRODUCE (post-generate) | Agent hit stop token | Tool call, return, nudge, or kill |
| `onSettleReject` | SETTLE stall-break | Deferred result, no active siblings | Nudge or drop |
| `shouldExplore` | DISPATCH | Before each tool executes | Explore vs exploit retrieval mode |
| `onToolRetry` | DISPATCH | Tool threw `ToolRetryError` | Park-and-retry or fail the call |
| `onRecovery` | Kill + post-loop | Killed agent without result | Extract findings (with budget) or skip |
| `pressureThresholds` | Pool setup | Once | softLimit / hardLimit |
| `resetTick` | Tick start | Every tick | Reset trailing-stop flags |

The policy receives the agent's full state (status, tool history, result, token count), a live `ContextPressure` snapshot, and pool configuration; it returns an action; the pool executes it. Signatures are narrow by design — `(agent, pressure)` — because the policy is a class: time, cost, or other signals live on `this`.

The sections below walk the enforcement points in tick order, naming the hook each one consults.

## Enforcement points

### 1. Prefill gate (`ctx.spawn`) — and batch-mate reservation

When the orchestrator calls `ctx.spawn(...)`, the pool tokenizes the agent's suffix and checks `canFit`:

```typescript
const pressure = new ContextPressure(ctx, pressureOpts);
if (!pressure.canFit(reserved + suffixTokens.length)) {
  agent.branch.pruneSync();
  // pool:agentDrop reason=pressure_init
  throw new Error(`useAgentPool: cannot fit agent suffix under current pressure`);
}
```

Agents dropped at the prefill gate never emit `agent:spawn` — they disappear before generating a token. In parallel orchestration this means fewer agents run than requested; in chain, a mid-chain task can't start (surfacing as an orchestrator error); in nested pools, the outer agents' live branches consume the KV the inner gate measures against.

**The `reserved` term is the batch-mate reservation.** Admission is checked per spawn, but SPAWN-phase prefill is batched — every spawn admitted in a tick prefills together. The check therefore reserves the suffix tokens of already-admitted, not-yet-prefilled batch-mates: each spawn must fit *after* the ones before it, not merely against pre-batch headroom. Without the reservation, N individually-valid spawns can overrun the context in one prefill and every agent dies on its first turn; with it, oversubscription refuses the spawn loudly at admission time.

### 2. PRODUCE critical — the kill threshold (`shouldExit`)

At the start of each tick the per-agent PRODUCE loop consults `policy.shouldExit(agent, pressure)`, defaulting to `pressure.critical` when the hook is absent:

```typescript
for (const a of agents) {
  if (a.status !== 'active') continue;

  const policyExit = policy.shouldExit?.(a, pressure);
  if (policyExit ?? pressure.critical) {
    // pool:agentDrop reason=pressure_critical (or policy_exit)
    yield* recoverInline(a, policy, ctx, store, tw, ...);
    a.transition('idle');
    continue;
  }
  // ...produceSync + accumulate + commit
}
```

`DefaultAgentPolicy.shouldExit` returns `true` when `pressure.critical` fires or the time `hardLimit` is exceeded — **at most one kill per tick** (the `_killedThisTick` trailing-stop flag). Other agents survive to the next tick, where the just-killed agent's pruned branch gives them recovered headroom. If pressure stays critical, another agent is killed next tick.

**Recovery runs *before* the idle transition — the ordering is load-bearing.** `PoolContext.waitFor` resolves on the status signal; transitioning to idle first would let the orchestrator resume and start prefilling the next task while `recoverInline` is still decoding on the same `llama_context` — concurrent native calls, SEGV. Kill → recover → *then* idle.

**Terminal-tool protection is narrowed to the graceful zone.** When `agent.currentTool === terminalToolName` (the partial parser detected the model mid-emission of `report`) AND `!pressure.critical`, `shouldExit` returns `false` — an agent actively generating its report is protected from time-limit and custom kills. Once `pressure.critical` fires, protection yields: the agent must release KV before the next decode OOMs. This narrowing is what prevents the runaway-prose-into-terminal-tool pattern from exhausting the context. (`agent.currentTool` comes from `Agent.observe()` — a lenient partial parse on each generated token until the tool name appears.)

### 3. PRODUCE at stop token — `onProduced`, guards, and the nudge

When an agent hits a stop token, `parseChatOutput` extracts content and tool calls, and `policy.onProduced` decides what actually happens. Five possible actions: `tool_call` (dispatch it), `return` (terminal tool intercepted — extract `result`, mark idle), `nudge` (inject an error message as a fake tool result), `idle` (drop), `free_text_return` (prose without a tool call, captured as the result if the agent did work).

`DefaultAgentPolicy`'s decision flow:

```
parsed output has tool call?
  ├─ No → free_text_return (if agent has done work) or idle
  │
  ├─ Terminal tool? (e.g. report)
  │   ├─ Not under pressure + < minToolCallsBeforeReturn → nudge ("use tools first")
  │   └─ Under pressure OR sufficient work → return
  │
  ├─ Guard rejected? → nudge (with the guard's message)
  │
  ├─ Over budget? (turns >= maxTurns OR headroom < 0 OR time softLimit)
  │   ├─ Terminal tool configured + toolCallCount > 0 + !critical → nudge with word budget
  │   └─ Otherwise → idle (drop: maxTurns or pressure_softcut)
  │
  └─ All checks pass → tool_call
```

**Guards run before budget checks — deliberately.** A stuck agent that's over budget AND repeating a duplicate query gets the guard's specific message ("this query was already searched"), which names the actual problem, instead of a generic turn-limit nudge.

**Tool guards** are declarative rejection rules:

```typescript
interface ToolGuard {
  tools: string[] | '*';   // '*' sees every call
  reject: (args, lineageHistory, agent, toolName, config) => boolean;
  message: string;
}
```

`defaultToolGuards`, in order:

1. **authGuard** (`'*'`, framework-injected, runs first) — denies a `protected` tool the session holds no grant for. Fail-closed; emits `tool:authReject` with the agent's full lineage history for injection forensics. Open tools never consult it. See [AgentApp security](/build-an-app/app-security).
2. **Fetch dedup** — rejects `fetch_page` for a URL already fetched anywhere in the agent's *lineage* (self + ancestors, via `walkAncestors`).
3. **Search dedup** — rejects `web_search` for a query (case-insensitive) already searched in the lineage.

Replace or extend via `guards` (replaces all) / `extraGuards` (appends). Lineage vs local history matters when writing custom guards: a research-before-delegation guard must check *local* history, or a child agent inherits its parent's record and can delegate on turn 0 without doing any work.

**The nudge mechanism is stateless** — no escalation, no nudge count. The pool serializes the message as `{ "error": "..." }`, prefills it as a tool result, and resets the grammar. If the agent calls another non-terminal tool while still over budget, it's nudged again (one nudge per tick — `_nudgedThisTick`). There is no "second offense" kill; the hard kill is `shouldExit`'s job. The messages:

| Source | Message |
|--------|---------|
| KV pressure (PRODUCE) | "KV memory pressure — report your findings now within `{budget}` words." |
| Turn limit (PRODUCE) | "Turn limit reached — report your findings now within `{budget}` words." |
| Time budget (PRODUCE) | "Time limit reached — report your findings now within `{budget}` words." |
| SETTLE stall-break | "Tool result too large for remaining KV. Report your findings now within `{budget}` words." |
| Insufficient work | "You must use tools before submitting results." |
| Tool guard | The guard's `message` field |

The `{budget}` is `pressure.remaining - pressure.hardLimit` converted to **words, not tokens**: `floor(tokens * 0.7 / 10) * 10`, minimum 10. Tokenizers vary across models but words are universal in training data, and the 0.7 ratio + floor-to-10 under-advertise the ceiling so the model stays within decode headroom even if its word-to-token ratio differs. If the agent overshoots anyway, `pressure.critical` fires next tick and recovery takes over in the reserve; if it obeys, the voluntary return lands cleanly. Either way, findings land.

Terminal tools always pass through without nudging — an agent that wants to report is never blocked.

### 4. SETTLE deferral — tool result gating

After DISPATCH executes tools, the next tick's SETTLE drains the results, pressure-gated. SETTLE never kills directly — items that don't fit are **deferred** until headroom recovers:

```typescript
const settlePressure = new ContextPressure(ctx, pressureOpts);
let headroom = settlePressure.headroom;

for (const item of items) {
  if (item.prefillTokens.length > headroom) {
    deferred.push(item);  // retry next tick
    continue;
  }
  prefillPairs.push([a.branch, item.prefillTokens]);
  headroom -= item.prefillTokens.length;  // local accounting — cellsUsed updates only after store.prefill()
}
```

Tool results are the largest KV consumers — a `web_search` result can be 2,000+ tokens, a `fetch_page` result 6,000+. SETTLE deferral is the primary backpressure mechanism during active research.

- **Staggered exit (parallel).** Deferred items persist across ticks. Headroom recovers when a sibling reports and its branch is pruned (`pruneOnReturn: true`); next tick the deferred result fits. One agent's big result is absorbed by another's completion.
- **Spine growth (chain).** Each task's findings accumulate on the spine via `extendSpine` (typically 500–1,500 tokens per task); later tasks start with less headroom. Size limits for the *last* task's constraints, not the first's.
- **DAG** is a hybrid — dependent nodes inherit extended spine; independent siblings stagger like parallel.

### 5. SETTLE stall-break — `onSettleReject`

If `deferred` has items but no active siblings can free KV (everyone is `awaiting_tool` or idle), the stall-breaker consults `policy.onSettleReject` per item:

```typescript
type SettleAction =
  | { type: 'nudge'; message: string }   // replace the oversized result with a compact payload
  | { type: 'idle'; reason: IdleReason } // drop; recovery extracts
```

`DefaultAgentPolicy` returns `nudge` when a terminal tool is configured and the agent has made at least one tool call — same word-budget message as the PRODUCE nudge. The nudge replaces the oversized result; the agent's next generation is reaction-to-nudge (report, or retry with a narrower query). Three distinct outcomes in the trace:

- nudge fits → no drop, agent continues;
- policy returned `idle`, or the nudge payload itself doesn't fit → drop `pressure_settle_reject`;
- hook absent → drop `settle_stall_break` (legacy fallback).

Unlike the PRODUCE trailing stop, the stall-break can drop or nudge **multiple** agents on the same tick — by this point there is no alternative left to waiting.

### 6. DISPATCH — explore/exploit and retry parking

**`shouldExplore`** runs before each tool executes and sets `ToolContext.explore`. `true` (the default with headroom) → content tools score retrieved chunks against the tool's own query only. `false` (exploit) → dual scoring via `scoreRelevanceBatch` — `min(toolQueryScore, originalQueryScore)` — tightening focus at the cost of serendipitous discovery. `DefaultAgentPolicy` checks two independent axes; **either** crossing its threshold flips to exploit:

- **Context axis**: `percentAvailable / 100 > shouldExplore.context` (default 0.4) required for explore.
- **Time axis**: `elapsed / timeSoftLimit < shouldExplore.time` (default 0.5) required for explore — only when `budget.time.softLimit` is set.

The flip is non-monotonic: `percentAvailable` recovers as branches are pruned, and an agent in exploit mode flips back to explore when siblings free KV. Intentional — a quality knob, separate from the lifecycle. Tool-side behavior is documented in [Sources & retrieval](/build-an-app/sources-and-retrieval); `policy.setExploitMode(true)` (on `DefaultAgentPolicy`, not the interface) is the harness-driven override.

**`onToolRetry`** handles transient backend failures. A tool that's rate-limited throws `ToolRetryError` with a `retryAfterMs` hint instead of returning an error result. The pool **parks** the agent — it stays `awaiting_tool`, skipped by PRODUCE at zero cost: no turns, no tokens, no KV — and re-dispatches the *same call, same callId* after the delay. The model's context never records the failure: a transient condition the model can't act on doesn't belong in its KV.

```typescript
onToolRetry(agent, tool, error, attempt): ToolRetryAction
// { type: 'retry', afterMs? }  — park and re-dispatch (afterMs overrides the tool's hint)
// { type: 'fail', message? }   — settle a directive result; the model pivots
```

`DefaultAgentPolicy` retries while `attempt <= maxToolRetries` (default 1). Exhaustion is **not** a `tool_error` — the agent receives an honest tool result ("currently unavailable — use other sources or proceed with your findings") and decides its next move with its evidence intact. Observability: `agent:tool_retry` events and `tool:retry` traces, so a waiting agent reads as waiting, not hung.

## Recovery extraction

When an agent is killed by pressure (or ends idle without a result), the pool extracts its accumulated findings **inline** before the tick loop continues — one agent sacrificed per critical tick, findings extracted, KV freed, remaining agents continue.

    alt="Flowchart showing the recovery extraction protocol: pressure check, eager grammar, single-agent produce/commit loop."
  />
</Frame>

Recovery is **in-place — no fork**. The agent's full KV context (system prompt, tool results, reasoning) is already physically in the cache; the extraction prompt is prefilled directly onto the dying agent's own branch.

### The protocol

1. **Policy check**: `policy.onRecovery(agent, pressure)`. `DefaultAgentPolicy` applies the confabulation guard first — `agent.tokenCount >= minTokens` (default 100) and `agent.toolCallCount >= minToolCalls` (default 2). An agent that generated 20 tokens and made 1 call would fabricate findings; `skip` prunes it instead.
2. **Budget computation**: `budget = max(50, pressure.remaining - prefillOverhead(150) - batchBuffer(512))`, converted to words and rendered into the prompt's Eta template — authors write `<%= it.budget %>` where the budget should surface; prompts without the tag render unchanged.
3. **Eager grammar**: `jsonSchemaToGrammar({ result: string })`, set via `setGrammar` — constrained from token 0. During normal generation agents use *lazy* grammar (free generation until a `<tool_call>` trigger); recovery is the system extracting findings, not asking the model to report. No tool calls possible, no model choice.
4. **Prefill + generate**: the rendered prompt prefills onto the agent's branch; a single-agent produce/commit loop runs to the stop token, inside `scoped()` error containment — a decode failure tears down cleanly without crashing the pool.
5. **Parse + diagnostics**: `pool:recoveryProduce` (token count + output length), then exactly one of `pool:recoveryReturn` (findings captured via `setResult(parsed.result, 'recovery')`) or `pool:recoveryFailed` (with `reason` — `parse_error` / `no_result_field` / `scope_error` — and a 200-char `outputExcerpt`). No silent recovery failures.
6. **Prune**: the branch's unique cells free; remaining agents see the headroom next snapshot.

A minimal recovery prompt:

```eta
You are a research reporter. The agent has been terminated due to budget
constraints and must deliver findings now.

Call the report tool with the most important findings from the research above.
Prioritize: 1. direct quotes and specific evidence  2. key conclusions with
source URLs  3. supporting context (only if budget allows).

IMPORTANT: You have approximately <%= it.budget %> words. Be concise — if you
exceed the budget your report will be cut off.
---
Report your findings.
```

### Recovery and `hardLimit`

The reserve must fit: extraction prompt prefill (~100–200 tokens) + the `{"result": "..."}` generation (200–500 for a typical report) + the native batch workspace (~512 cells). Default `hardLimit = 512` satisfies the invariant but is tight for verbose reports; bump to 1024+ if you expect long extractions. The recovery pressure check uses raw remaining capacity, not `critical` — the reserve was set aside precisely for this moment.

### What recovery does NOT do

- Recovery tokens emit `agent:produce` events (visible in a TUI stream) but are **not** accumulated into `rawOutput` and not written as `agent:turn` trace events.
- `agent.finalize()` is not called — recovery runs its own produce/commit loop outside the normal tick flow.
- The grammar-constrained JSON may contain `<think>` tags or tool-call XML as literal string content inside `result`. Strip structural markers before prefilling extracted text onto a spine if you want clean spine content.
- Provenance is metadata only: recovered findings carry `resultSource: 'recovery'` and emit `agent:recovered` (not `agent:return`); no downstream code branches on it.

### One-shot guard and scope

If agents end idle without results after the tick loop exits (killed by `maxTurns` or `free_text_stop` — paths that don't recover inline), a single post-loop pass extracts from any unrecovered agents (`recoveryAttempted` fires once per pool). Recovery is per-pool: in chain pipelines, recovered findings ride `extendSpine` onto the spine exactly like voluntary returns — subsequent tasks inherit them via KV.

Production note: reasoning.run's recon stage is this same mechanism with a recon-shaped policy — `ReconPolicy` hard-caps turns via `shouldExit`, and force-kills recover with a `PREFLIGHT_RECOVER` prompt.

## `cellsUsed` tracks live pressure

<Note>
`cellsUsed` increments on every `decode_each` (COMMIT, +1 per branch) and `decode_scatter` (SETTLE prefill, +tokens per branch). It **decrements** when a branch is pruned — each branch subtracts its unique cells (`position - fork_head`) on release. It resets on bulk operations: `retainOnly` sets it to the winner's position, `drain` resets to zero.

`cellsUsed` can exceed `nCtx` — there is no pre-decode guard in the native layer. Protection is entirely in the JS layer via `ContextPressure.critical` + the `hardLimit >= nBatch` invariant. On `llama_decode` failure, `cellsUsed` is NOT rolled back, but the branch's sampler, grammar, and metrics state IS rolled back via native snapshot/restore.
</Note>

`remaining` and `percentAvailable` **recover** as branches are freed. When an agent reports and its branch is pruned, the freed cells are visible to the next snapshot — surviving agents see increased headroom, and `percentAvailable` can rise back above the `shouldExplore.context` fraction, flipping them back to explore mode.

## `DefaultAgentPolicy` configuration

```typescript
import { DefaultAgentPolicy } from "@lloyal-labs/lloyal-agents";

const policy = new DefaultAgentPolicy({
  minToolCallsBeforeReturn: 2,                  // confabulation gate on returns
  extraGuards: [],                              // appended to defaults (guards: replaces)
  shouldExplore: { context: 0.4, time: 0.5 },   // explore/exploit axes
  budget: {
    context: { softLimit: 1024, hardLimit: 512 },     // KV tokens remaining
    time: { softLimit: 480_000, hardLimit: 600_000 }, // ms — unset by default
  },
  recovery: {
    prompt: { system: "...", user: "Report. <%= it.budget %> words." },
    minTokens: 100,
    minToolCalls: 2,
  },
  terminalToolName: "report",  // enables terminal-tool kill protection
  maxToolRetries: 1,
});
```

| Option | Default | Effect |
|--------|---------|--------|
| `minToolCallsBeforeReturn` | 2 | Returns with fewer non-terminal calls get nudged (unless under pressure) |
| `guards` / `extraGuards` | defaults / `[]` | Replace / append tool guards |
| `shouldExplore.context` | 0.4 | Min fraction of KV remaining for explore mode |
| `shouldExplore.time` | 0.5 | Max fraction of time softLimit consumed before exploit |
| `budget.context.softLimit` | 1024 | KV floor that triggers nudges/deferral |
| `budget.context.hardLimit` | 512 | Kill floor + recovery reserve; must be ≥ `nBatch` |
| `budget.time.*` | unset | No time budget unless configured |
| `recovery.*` | unset | No extraction unless configured |
| `terminalToolName` | unset | Protects mid-report agents from non-critical kills |
| `maxToolRetries` | 1 | Park-and-retry attempts per call on `ToolRetryError` |

**Time budget scope**: `_startTime` is per-policy-instance and elapsed time is measured per-agent from its first activation — but the same instance is shared with nested pools spawned by `DelegateTool`, so a "10-minute research budget" applies at every depth. Create separate instances for stages with different constraints (research vs synth). `DelegateTool.createPolicy` exists precisely so each delegation gets a fresh clock.

### Writing a custom policy

```typescript
class MyPolicy implements AgentPolicy {
  onProduced(agent, parsed, pressure, config): ProduceAction { /* ... */ }
  onSettleReject(agent, resultTokens, pressure, config): SettleAction { /* ... */ }
  shouldExplore(agent, pressure) { return pressure.percentAvailable > 50; }
  shouldExit(agent, pressure) { return pressure.critical; }
  onRecovery(agent, pressure): RecoveryAction {
    if (agent.tokenCount < 100) return { type: "skip" };
    return { type: "extract", prompt: { system: "...", user: "..." } };
  }
  get pressureThresholds() { return { softLimit: 2048, hardLimit: 1024 }; }
}

yield* agentPool({ orchestrate, tools, policy: new MyPolicy() });
```

## Sizing thresholds

For single-level pools with default `nBatch=512`:

| Policy | softLimit | hardLimit | Rationale |
|---|---|---|---|
| Baseline (default) | 1024 | 512 | Minimum invariant-satisfying config |
| Research with recovery | 1024–2048 | 1024 | Generous budget for recovery extraction |
| Chain pipeline (4+ tasks) | 2048 | 1024 | Spine accumulates — size for the last task |
| Nested / recursive pools | 3072 | 1024 | Outer pool reserves the inner pool's full lifecycle (~500 tokens × inner agent count) |

Monitor with `trace: true` — pressure prints at each tick, and recovery emits its diagnostic pair:

```
[PRODUCE] SOFT_LIMIT remaining=2200 headroom=-104 cellsUsed=14184 nCtx=16384
[SETTLE] DEFER read_file:1059 > headroom=-104
[SETTLE] NUDGE read_file: 33tok (replacing 1059)
[RECOVERY] prefilling agent=3, 143 tokens, remaining=1024
pool:recoveryProduce agent=3 tokenCount=218 outputLength=1204
pool:recoveryReturn  agent=3 resultLength=1180
```

## Pressure decision flow

## Kill / nudge decision matrix

The decision to kill or nudge happens at multiple phases, each with its own inputs and side-effect chain. The matrix enumerates every decision point.

### Phases

| Phase | Where | When |
|---|---|---|
| SPAWN | `ctx.spawn` prefill gate | Before agent runs |
| PRODUCE top-of-tick | `policy.shouldExit` | Each tick, before `produceSync` |
| PRODUCE at `isStop` | `policy.onProduced` | Agent emits stop token |
| SETTLE normal | `settle()` fit check | Each tick when items present |
| SETTLE stall-break | `policy.onSettleReject` | Deferred items + no active siblings |
| Pool close | cleanup loop | After tick loop exits |

### Decisions

| Phase | Condition | Outcome | Recovery? |
|---|---|---|---|
| SPAWN | suffix + reserved > headroom | drop `pressure_init`, `ctx.spawn` throws | no |
| PRODUCE top | `pressure.critical` + no `shouldExit` hook | drop `pressure_critical` | yes |
| PRODUCE top | `pressure.critical` + `_killedThisTick` already set | skip (one kill per tick) | no |
| PRODUCE top | `pressure.critical` + `currentTool === terminalToolName` | drop `pressure_critical` (protection yields under critical) | yes |
| PRODUCE top | `!pressure.critical` + `currentTool === terminalToolName` | protected, no drop | n/a |
| PRODUCE top | `shouldExit` returns `true` (time limit, custom) | drop `policy_exit` | yes |
| PRODUCE `isStop` | tool call + `headroom < 0` + terminal tool configured + `toolCallCount > 0` | nudge (`_nudgedThisTick`) | no |
| PRODUCE `isStop` | tool call + `headroom < 0` + no terminal tool | drop `pressure_softcut` | no (post-loop only) |
| PRODUCE `isStop` | `turns >= maxTurns` + terminal tool configured | nudge "Turn limit reached" | no |
| PRODUCE `isStop` | `turns >= maxTurns` + no terminal tool | drop `maxTurns` | no (post-loop only) |
| PRODUCE `isStop` | terminal tool call | report captured (no drop) | n/a |
| SETTLE normal | item fits | prefill, transition to `active` | n/a |
| SETTLE normal | item doesn't fit + any active sibling | defer to next tick | n/a |
| SETTLE normal | item doesn't fit + no active siblings | invoke stall-break (below) | — |
| SETTLE stall-break | `onSettleReject` → `nudge` + nudge fits | replace item with nudge payload, agent continues | no |
| SETTLE stall-break | `onSettleReject` → `nudge` + nudge doesn't fit | drop `pressure_settle_reject` | yes |
| SETTLE stall-break | `onSettleReject` → `idle` | drop `pressure_settle_reject` | yes |
| SETTLE stall-break | hook absent | drop `settle_stall_break` | yes |
| Pool close | idle without result + `onRecovery` → `extract` | `recoverInline` only (no drop event) | yes |
| Pool close | idle without result + `onRecovery` → `skip` | branch prune only | no |

### Hidden state interactions

- **`_killedThisTick`** gates `shouldExit` — one kill per tick in critical. Does NOT apply to the SETTLE stall-break, which can drop multiple agents in one tick.
- **`_nudgedThisTick`** gates the PRODUCE over-budget nudge — one per tick. Does NOT apply to the stall-break, which can nudge once per deferred item.
- **Terminal-tool protection** is only checked in `DefaultAgentPolicy.shouldExit`. Stall-break drops and pool-close cleanups do NOT check `currentTool`.
- **PRODUCE `isStop` drops (`maxTurns`, `pressure_softcut`) do NOT run inline recovery.** Those agents stay idle with live branches; the pool-close pass extracts them.

## Drop reasons

| Reason | Origin | Meaning |
|---|---|---|
| `pressure_init` | `ctx.spawn` prefill gate | Agent suffix doesn't fit under softLimit |
| `pressure_critical` | PRODUCE `shouldExit` | `remaining < hardLimit`; recovery runs in reserve |
| `pressure_softcut` | PRODUCE over-budget | Non-terminal over-budget + no terminal tool available |
| `pressure_settle_reject` | SETTLE stall-break (policy-driven) | Policy returned `idle` or a nudge that can't fit |
| `settle_stall_break` | SETTLE stall-break (fallback) | `onSettleReject` hook absent |
| `time_exceeded` | PRODUCE `shouldExit` | Wall-clock hardLimit crossed |
| `policy_exit` | PRODUCE `shouldExit` (custom) | Policy returned `true` not driven by pressure |
| `maxTurns` | PRODUCE at `isStop` | Turn count ≥ maxTurns, no terminal tool |
| `tool_error` | DISPATCH | Tool's `execute()` threw (non-retryable) |

Every drop reason except `pressure_init` is followed by a recovery attempt (inline or post-loop) whose outcome is visible in `pool:recoveryReturn` or `pool:recoveryFailed` — no silent data loss.
