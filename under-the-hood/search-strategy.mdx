The reranker is RIG infrastructure, not a fetch optimization. The same cross-encoder model drives **four distinct scoring roles** across an agent's lifetime — content entailment, relevance under pressure, similarity for echo detection, and a floor gate that decides whether a result is worth surfacing at all. This page is the reference for the scoring shape, the score formula, and where each method fires in the tick loop.

If you're authoring an App, [Sources & retrieval](/build-an-app/sources-and-retrieval) is the higher-level walkthrough — the page you're reading is for harness authors, framework debuggers, and anyone designing custom retrieval flows.

## The score formula: logit-diff, range unbounded

The reranker returns `logit(yes) − logit(no)` per (query, document) pair — the raw logit difference, **not** a softmax probability. Sign carries meaning:

| score | interpretation |
| --- | --- |
| **positive** | the model would emit "yes" rather than "no" if forced — relevant |
| **zero** | indifferent |
| **negative** | "no" — irrelevant |
| **magnitude** | confidence |

`SearchTool` and `EntailmentScorer.shouldProceed` both threshold on this scale. The default floor on `Source._entailmentFloor` is `0` — a hit passes when the cross-encoder *leans yes*. Subclasses can tighten (`2` for "confident yes" in noisy corpora) or loosen (negative floors when sparse data means leaning-no hits are still worth investigating).

<Note>
The prior scoring form returned `P(yes | yes_or_no) ∈ (0,1)` via softmax over the yes/no logits, with a `0.25` default floor. The softmax compressed small logit gaps into extreme probabilities (gap of 5 → 0.993; 10 → 0.99995), saturating top-K ordering and erasing confidence differences in dense result sets. Calibration data (`reasoning.run/scripts/inspect-rerank.mjs`) showed:

- median(relevant) − median(irrelevant) jumped **0.0453 → 3.66** (×80)
- top-5 stddev per query **0.05–0.5 → 2.5–6.5** (saturation eliminated)
- on the canonical fixture, AUC moved from per-query 0.625-noise to consistent ≥0.85 for API-identifier queries

The logit-diff change is in `packages/sdk/src/Rerank.ts`; the floor recalibration is in `packages/agents/src/source.ts` (default `_entailmentFloor = 0`).
</Note>

## Three queries in play

The four scoring methods aren't variants on one concept — they operate on three *distinct* queries the framework keeps separate by design:

| Concept | Scope | Field name in code |
| --- | --- | --- |
| **Tool query** | Per tool call | `localQuery` param of `scoreRelevanceBatch` |
| **Agent task** | Per agent lifetime | `reference` param of `scoreSimilarityBatch` |
| **Original query** | Per research invocation | Captured in closure by `createScorer` |

Conflating these produces wrong scores. When you add a new scoring method or trace event, use the field names from this table.

## The four scoring roles

```ts
export interface EntailmentScorer {
  scoreEntailmentBatch(texts: string[]): Promise<number[]>;
  scoreRelevanceBatch(texts: string[], localQuery: string): Promise<number[]>;
  scoreSimilarityBatch(reference: string, texts: string[]): Promise<number[]>;
  shouldProceed(score: number): boolean;
}
```

### 1. `scoreEntailmentBatch` — content vs. original query

Used at **content boundaries** to keep retrieved content on the original research intent. Each text is scored against the **original query** (captured in closure when the harness called `Source.createScorer(originalQuery)`).

**Fires at:** SETTLE phase, when a tool result lands and the framework is about to prefill it into the agent's KV. A failing entailment gate causes the result to be summarized or dropped before it pollutes the KV at the call position.

### 2. `scoreRelevanceBatch` — dual gate under pressure

Used in **exploit mode** when KV pressure tightens focus. Scores each text twice — against the per-call tool query *and* against the original query — and returns `min(toolQueryScore, originalQueryScore)`. A chunk has to clear both bars.

**Fires at:** inside the tool (e.g. `SearchTool.execute`) when `ToolContext.scorer` is set and `explore` is `false`. The pool decides which mode the tool runs in based on `KV-pressure`; the tool obeys.

### 3. `scoreSimilarityBatch` — echo detection at delegation

Used at **delegation boundaries** to catch an agent re-issuing work that paraphrases its own task. Scores candidate sub-tasks against an arbitrary `reference` — typically the parent agent's task.

**Fires at:** `DelegateTool.execute` (and similar fan-out tools), before spawning the sub-agent. A high similarity score against the parent's own task means the sub-agent would just paraphrase its parent — short-circuit instead of forking.

### 4. `shouldProceed` — the floor gate

The simple boolean: does this score clear `_entailmentFloor`? Used everywhere a method-specific gate needs a uniform threshold.

```ts
shouldProceed(score: number): boolean {
  return score >= this._floor;  // default _floor = 0
}
```

## Explore vs exploit

The pool tells the tool which mode to score in via `ToolContext.explore`:

- **Explore (default, headroom available).** `SearchTool` scores chunks against the **tool query** alone — agents discover bridging content adjacent to the answer even when the original query wouldn't surface it.
- **Exploit (under pressure).** `SearchTool` calls `context.scorer.scoreRelevanceBatch` — `min(toolQueryScore, originalQueryScore)` — tightening focus at the cost of serendipitous discovery.

The agent never knows which mode it's in; `DefaultAgentPolicy.shouldExplore` decides per dispatch. See [Agent policy](/under-the-hood/agent-policy) for the threshold and [KV pressure](/under-the-hood/kv-pressure) for what counts as "under pressure."

## SearchTool envelope

`SearchTool` returns a structured envelope so the agent has an honest "nothing matched" signal instead of getting fed garbage shaped as evidence:

```ts
type SearchResult = {
  hits: ScoredChunk[];               // hits ≥ thresholdScore, sorted desc
  thresholdScore: number;            // the floor applied (default 0)
  totalScored: number;               // total corpus chunks the reranker saw
  topRejected: ScoredChunk[];        // top 3 sub-floor chunks; only populated
                                     // when `hits` is empty
};
```

The agent sees `{ hits: [], thresholdScore: 0, totalScored: 484, topRejected: [...] }` and knows the corpus has nothing the model would call relevant. That's a real "best I could find, but the model said no to all of them" signal. Previously the tool returned top-K by score regardless of relevance — irrelevant queries got `0.999`-saturated noise that the agent processed as evidence.

## Cross-request multiplex

A single reranker context handles concurrent scoring via `Rerank._drain` and `_fillGroup`:

- Multiple `score()` calls enqueue into `_pending` and the drain loop pulls work via `_fillGroup`, packing across requests up to `nSeqMax` slots per GPU dispatch.
- Two concurrent `scoreBatch` calls on the same context race (`scoreBatch` bypasses the multiplex layer). `EntailmentScorer.scoreRelevanceBatch` uses `scoreBatch` internally — so callers must serialize relevance scoring per reranker context.

The agent pool's serial-DISPATCH invariant covers this today by ensuring one `tool.execute` runs at a time per pool. If you write a custom scoring path that lives outside DISPATCH, route through the iterator (`reranker.score(...)`) rather than `reranker.scoreBatch(...)` to stay multiplex-safe.

## Calibration discipline

When tuning a reranker for a new corpus:

1. **Run the calibration probe** (`reasoning.run/scripts/inspect-rerank.mjs`) against a fixture set with ground-truth labels: relevant chunks + off-topic chunks + API-identifier queries.
2. **Check** median(relevant) − median(irrelevant) ≥ 0.30 (the loose target; the production fix moved this to 3.66).
3. **Check** top-5 stddev per query > 1.0 (saturation should be visibly gone).
4. **Threshold sweep** — at floor `0`, `0.5`, `2.0`, `5.0`, what fraction of relevant/irrelevant chunks pass? Pick the lowest floor that keeps the irrelevant pass-rate near zero.

Floor changes propagate as a subclass override on `Source._entailmentFloor`, not as a global flag. App authors who know their corpus's characteristics can tighten without changing framework defaults.
