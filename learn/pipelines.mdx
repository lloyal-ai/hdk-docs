A pipeline is a sequence of generator stages. Each stage `yield*`s into a framework primitive — `agent`, `agentPool`, `diverge`. The output of one stage feeds the next.

Composition is code-first: stages are TypeScript generators with Effection scopes, wired together with `yield*`. There is no separate pipeline DSL or visual builder. Within a single `agentPool` call, structured **orchestrators** — `parallel`, `chain`, `fanout`, `dag`, `dagWithEvents`, `reduce` — describe how that pool's agents relate to one another (fan-out, sequential, multi-parent dependency graphs). They're optional building blocks; the contract is one function — `Orchestrator = (ctx: PoolContext) => Operation<void>` — and the built-ins are 10–30 line implementations of it.

## The pattern

Every stage is a generator function that takes input and produces output:

```typescript
function* plan(query: string): Operation<string[]> {
  const result = yield* agent({
    systemPrompt: PLAN_PROMPT,
    task: query,
    schema: planSchema,
  });
  return JSON.parse(result.rawOutput).questions;
}

function* research(questions: string[]): Operation<string[]> {
  const pool = yield* agentPool({
    orchestrate: parallel(
      questions.map(q => ({ content: q, systemPrompt: RESEARCH_PROMPT })),
    ),
    tools: [searchTool, fetchTool, reportTool],
    terminalTool: "report",
  });
  return pool.agents.map(a => a.result).filter(Boolean);
}

function* synthesize(findings: string[], query: string): Operation<string> {
  const pool = yield* agentPool({
    orchestrate: parallel([
      { content: buildSynthPrompt(findings, query), systemPrompt: SYNTH_PROMPT },
    ]),
    tools: [...groundingTools, reportTool],
    terminalTool: "report",
  });
  return pool.agents[0]?.result || "";
}
```

Compose them in sequence:

```typescript
function* handleQuery(query: string): Operation<string> {
  const questions = yield* plan(query);
  const findings = yield* research(questions);
  const answer = yield* synthesize(findings, query);
  return answer;
}
```

That's a pipeline. Each stage runs to completion. Resources are cleaned up between stages — `agentPool` prunes branches when it returns, freeing KV for the next stage.

<Tip>
Stages are independent generators. You can test them in isolation, reorder them, skip them conditionally, or add new ones. The composition is plain code — `if` statements, loops, early returns.
</Tip>

## The building blocks

Three primitives implement the work inside a stage:

<CardGroup cols={3}>
  <Card title="agent()" icon="wand-magic-sparkles">
    Single agent. Plan decomposition (with `schema`), JSON extraction, evaluation. Returns an Agent with result and raw output.
  </Card>
  <Card title="agentPool()" icon="users">
    Multi-agent pool with tools and an `orchestrate` argument that picks how the pool's agents relate (parallel / chain / fanout / dag).
  </Card>
  <Card title="diverge()" icon="code-branch">
    Multi-branch perplexity selection. Fork N attempts, generate in parallel, select the lowest-perplexity output. Verification and confidence checking.
  </Card>
</CardGroup>

`reduce()` is a stage-level combinator (covered in [Multi-source pipelines](#multi-source-pipelines-with-reduce) below) — it threads accumulated state across stages rather than describing relationships within one pool.

## Orchestrators

A pool's `orchestrate` argument selects how its agents relate. The SDK ships five composable shapes plus an escape hatch — pick the simplest one that fits your dependency pattern.

<CardGroup cols={2}>
  <Card title="parallel(specs)" icon="grip-lines-vertical">
    Fan-out. All agents fork from the shared root concurrently. Use when sub-tasks are independent.
  </Card>
  <Card title="chain(items, toStep)" icon="link">
    Sequential. Each step extends the spine before the next forks — later agents see earlier findings via KV attention. Use when later tasks build on earlier ones.
  </Card>
  <Card title="fanout(landscape, domains)" icon="diagram-project">
    Landscape pass + N domain agents from the post-landscape root. Use when one shared scouting step should anchor several specialized investigations.
  </Card>
  <Card title="dag(nodes)" icon="circle-nodes">
    Arbitrary acyclic graph with multi-parent dependencies. Lazy spawn on dependency resolution; independent nodes run in parallel. Use when the dependency shape isn't a chain or pure fan-out.
  </Card>
  <Card title="dagWithEvents(nodes, emit)" icon="bell">
    `dag` with a progress-event hook for TUI / telemetry. Same semantics, plus emit-on-spawn / emit-on-complete callbacks.
  </Card>
  <Card title="(custom)" icon="code">
    `Orchestrator = (ctx: PoolContext) => Operation<void>`. The built-ins are 10-30 line implementations of this. Write your own when none of the above fits.
  </Card>
</CardGroup>

## Example pipelines

### Research pipeline

Plan, dispatch agents across sources, evaluate findings for convergence, synthesize (with grounding tools when agents disagree), verify with perplexity sampling.

```typescript
function* researchPipeline(query: string): Operation<string> {
  // Stage 1: Decompose query into sub-questions
  const questions = yield* plan(query);

  // Stage 2: Research agents search across sources
  const res = yield* research(questions, sources);

  // Stage 3: Evaluate findings convergence — do agents agree?
  const eval = yield* evaluateFindings(res.agentFindings);

  // Stage 4: Synthesize — grounding tools only when conflicts detected
  const conflicts = eval.conflicts.length ? eval.conflicts : undefined;
  const answer = yield* synthesize(res.agentFindings, res.sourcePassages, query, conflicts);

  // Stage 5: Verify with N parallel samples
  const verified = yield* verify(answer, query);

  return verified.answer;
}
```

### Code review pipeline

Parse a diff, dispatch agents per file, collect issues, produce a summary.

```typescript
function* codeReview(diff: string): Operation<string> {
  // Stage 1: Extract changed files from diff
  const result = yield* agent({
    systemPrompt: "Extract changed files.",
    task: diff,
    schema: filesSchema,
  });
  const files = JSON.parse(result.rawOutput);

  // Stage 2: One agent per file — each searches the codebase for context
  const reviews = yield* agentPool({
    orchestrate: parallel(
      files.map(f => ({
        content: `Review changes to ${f.path}:\n${f.diff}`,
        systemPrompt: REVIEW_PROMPT,
      })),
    ),
    tools: [grepTool, readFileTool, reportTool],
    terminalTool: "report",
  });

  // Stage 3: Synthesize per-file reviews into summary
  const summary = yield* synthesize(reviews.agents.map(a => a.result), diff);
  return summary;
}
```

### Support pipeline

Classify intent, search knowledge base, draft response, check confidence.

```typescript
function* supportTicket(ticket: string): Operation<{ response: string; confident: boolean }> {
  // Stage 1: Classify — grammar-constrained to valid categories
  const classifier = yield* agent({
    systemPrompt: "Classify support tickets.",
    task: ticket,
    schema: categorySchema,
  });

  // Stage 2: Search knowledge base for relevant articles
  const pool = yield* agentPool({
    orchestrate: parallel([{ content: ticket, systemPrompt: KB_PROMPT }]),
    tools: [kbSearchTool, kbReadTool, reportTool],
    terminalTool: "report",
  });

  // Stage 3: Confidence check — low perplexity = model is confident
  const samples = yield* diverge({ prompt: draftPrompt(pool.agents[0]?.result, ticket), attempts: 3 });
  const confident = samples.best.ppl < 2.0;

  return { response: samples.best.output, confident };
}
```

## Conditional stages

Pipelines are plain code. Use `if` statements, early returns, and loops.

```typescript
function* handleQuery(query: string): Operation<QueryResult> {
  const plan = yield* classify(query);

  // Early return for clarification
  if (plan.intent === "clarify") {
    return { type: "clarify", questions: plan.questions };
  }

  // Passthrough for simple queries — skip decomposition
  const questions = plan.questions.length > 0
    ? plan.questions.map(q => q.text)
    : [query];

  const findings = yield* research(questions);
  const answer = yield* synthesize(findings, query);
  return { type: "done", answer };
}
```

<Note>
Cross-stage control flow is plain code: `if` for conditionals, `for` for loops, calling the stage again for retry. Within a stage, when one pool's agents form a dependency graph, reach for the structured orchestrators — `dag` for arbitrary acyclic deps, `chain` for sequential, `parallel` or `fanout` for fan-out shapes (see [Orchestrators](#orchestrators) below).
</Note>

## Findings convergence

When multiple agents investigate sub-questions, they may return contradictory findings. The findings convergence check surfaces real conflicts and separates them from complementary coverage:

```typescript
const eval = yield* evaluateFindings(agentFindings);

if (eval.conflicts.length) {
  // Real contradictions — synthesize with grounding tools
  const answer = yield* synthesize(findings, passages, query, eval.conflicts);
} else {
  // Agents agree — synthesize directly, no grounding needed
  const answer = yield* synthesize(findings, passages, query);
}
```

The eval also produces `observations` — cross-agent analysis (coverage gaps, complementary findings) that flow to the synthesizer as context without triggering grounding tools.

When conflicts are detected, the synthesizer receives source tools (search, read_file, fetch_page) to read the source material directly. It resolves each conflict by going to the text — not by splitting the difference between agents. When findings converge, the synthesizer gets `report` only — no wasted grounding.

<Note>
The prompt itself adapts to what's available. Using [Eta templates](/reference/prompt-composition), the system prompt only describes capabilities the agent actually has — no phantom "Source passages: (none)" sections. Conditionals in the template control which instructions, sections, and tools appear.
</Note>

## Multi-source pipelines with `reduce`

When a pipeline uses multiple sources, `reduce` folds across them sequentially, threading accumulated findings forward:

```typescript
import { reduce } from "@lloyal-labs/lloyal-agents";

const findings = yield* reduce(
  sources,
  { sections: [], questions },
  function*(acc, source, i) {
    const pool = yield* agentPool({
      orchestrate: parallel(
        acc.questions.map(q => ({ content: q, systemPrompt: sourcePromptFor(source) })),
      ),
      tools: [...source.tools, reportTool],
      terminalTool: "report",
    });

    const sections = [...acc.sections, ...collectFindings(pool)];

    // Between sources: bridge discoveries into context for the next source
    if (i < sources.length - 1) {
      const bridged = yield* bridge(sections, query);
      const enriched = questions.map(q => `${q}\n\nPrior discoveries:\n${bridged}`);
      return { sections, questions: enriched };
    }
    return { sections, questions: acc.questions };
  },
);
```

Each source is independent. The bridge stage distills findings from source N into context for source N+1. Later sources benefit from earlier research without re-doing it.

## KV lifecycle between stages

Each stage that uses `agent` or `agentPool` creates and destroys branches within its scope. When the call returns, branches are pruned and KV cells are freed.

```
Stage 1: plan          → agent uses ~500 KV cells → pruned
Stage 2: research      → agentPool uses ~8,000 cells → pruned
Stage 3: findings eval → agent uses ~300 cells → pruned
Stage 4: synthesize    → agentPool uses ~3,000 cells → pruned
Stage 5: verify        → agentPool uses ~1,500 cells → pruned
```

Stages don't compete for KV. Each stage gets the full context budget. This is why scoped cleanup is critical — branches are freed even if the stage throws.

<Tip>
When you need data to survive between stages, return it as plain values (strings, arrays, objects). KV state is ephemeral — it lives within a stage's scope. Findings, passages, and scores are the durable output.
</Tip>

## Session: state across turns

When the pipeline produces an answer, commit it to the Session so subsequent queries inherit the conversation context:

```typescript
yield* call(() => session.commitTurn(query, answer));
// Next agent/agentPool with `session` forks from the trunk
```
