## RIG vs RAG

Vanilla RAG retrieves once and generates once: query the vector DB, get top-k passages, inject them into the prompt, call the model. Retrieval and generation are separate phases. Modern variants — agentic RAG, self-RAG, multi-stage retrieval with reranking — already interleave the two, so the boundary is fuzzier than the textbook contrast suggests.

RIG is the name we use for our specific stack: an agentic retrieval loop where a cross-encoder reranker scores every retrieved candidate against the agent's *current* hypothesis before it's allowed into the working context. Agents generate reasoning, decide to search, process results, reason further, fetch content, form hypotheses, and search again with refined queries. Each retrieval decision is informed by everything the agent has already discovered — and the cross-encoder acts as a focal lens at every step, admitting only what scores.

The difference is observable in tool call inputs. A vanilla RAG system constructs search queries from the original user question. A RIG agent constructs queries from hypotheses formed during generation, and the reranker filters what those queries return before any of it reaches the decoder.

## Where This Fits

RIG is a name we coined; it isn't a new entry in the RAG taxonomy. The mechanism combines techniques the literature has separately named:

- **Agentic RAG** — an agent plans retrieval and executes it iteratively, with each call informed by prior results
- **Multi-stage retrieval with cross-encoder reranking** — a small classifier rescores retrieved candidates before they enter context
- **Iterative / self-refinement** — retrieval is gated by the agent's evolving hypothesis, not just the original query

What "RIG" labels is the opinionated stack: agentic retrieval + a cross-encoder acting as a focal lens at every retrieval step + shared KV-state across agents, tuned for small local models. The novelty is in the combination and the implementation — on-device cross-encoder, two-stage application against search results *and* page sections, running inside a shared KV spine — not the underlying pattern.

## The 4-Beat Pattern

Watch a RIG agent investigate a codebase:

```
grep(/memory leak/)            → 3 matches in 2 files
read_file(pool.ts L40-80)      → reads allocation logic, spots missing cleanup
search("resource cleanup on connection close")  → finds teardown handler
read_file(server.ts L120-155)  → discovers close handler never calls pool.drain()
grep(/drain|dispose|cleanup/)  → 8 matches, confirms drain exists but is unused
search("pool drain connection lifecycle interaction")  → targets the gap
report(findings)
```

The last search -- `"pool drain connection lifecycle interaction"` -- is the signature behavior. The agent read the allocation logic, discovered the drain method existed but was never called on close, and constructed a search targeting that specific interaction. This is multi-hop reasoning: not "search and report" but "search, form hypothesis, search for confirmation."

<Tip>
The pattern is not fixed at four beats. Agents naturally alternate between **narrow tools** (grep, read_file) and **broad tools** (search) based on what they find. Depth scales with `maxTurns`.
</Tip>

## Why It Emerges

This behavior is not prompted or engineered. It emerges from the [sequential dispatch model](/reference/concurrency#the-decision-boundary) — each tool result is fully prefilled before the agent generates its next action, creating a clean decision point where the model can form hypotheses from accumulated evidence.

<Note>
Under concurrent dispatch (where tool results arrive mid-generation), later queries are variations on the original question. Sequential dispatch produces progressively more specific queries informed by earlier results.
</Note>

## Depth Scales with Turns

At 2 turns, agents do single-shot retrieval. At 6 turns, they do 3-4 rounds of iterative refinement. At 20 turns, agents go deep -- following citation chains, cross-referencing claims, building evidence maps.

The quality difference is in the later tool call inputs. Early queries mirror the original question. Later queries target specific gaps the agent discovered during investigation.

## Sources

`@lloyal-labs/rig` provides two built-in `Source` implementations:

<CardGroup cols={2}>
  <Card title="CorpusSource" href="/reference/rig/corpus-source">
    Local files with grep, semantic search, read_file, and recursive research tools. Agents investigate a knowledge base by pattern matching and reading sections in context.
  </Card>
  <Card title="WebSource" href="/reference/rig/web-source">
    Web search via Tavily, page fetching with content extraction, and recursive web_research tools. Scratchpad extraction keeps KV pressure low.
  </Card>
</CardGroup>

Sources are composable. Use one, both, or [build your own](/reference/rig/custom-sources). When multiple sources run, they execute sequentially -- each source gets the full KV budget. After source N completes, its branches are pruned and KV is freed for source N+1.

Between sources, a [bridge](/reference/rig/bridges) structures discoveries into durable context for the next source.

## Install

```bash
npm i @lloyal-labs/rig @lloyal-labs/lloyal-agents @lloyal-labs/lloyal.node
```

The default `@lloyal-labs/rig` import is platform-agnostic (no Node.js or native dependencies). Node-specific exports (`createReranker`, `WebSource`, `CorpusSource`, `FetchPageTool`, `loadResources`, `chunkResources`) are available via `@lloyal-labs/rig/node` and require `@lloyal-labs/lloyal.node`.
