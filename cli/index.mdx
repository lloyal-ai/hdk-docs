`harness.dev` is the developer CLI for the HDK. It scaffolds new harnesses and apps, manages your publisher account on `apps.lloyal.ai`, ships your app to the signed-channel for review, and installs reviewed apps into any HDK harness.

## Install

```bash
npm install -g harness.dev
```

Or use `npx` directly without installing:

```bash
npx harness.dev <command>
```

`harness.dev` is Apache 2.0 — unrestricted. The framework runtime (`@lloyal-labs/lloyal-agents`, `@lloyal-labs/sdk`, `@lloyal-labs/rig`) is FSL-1.1-Apache-2.0 (see [Licensing](/licensing)).

## Command surface

| Command | Purpose | Details |
|---|---|---|
| `harness.dev <name>` | Scaffold a new harness | [Scaffolding](/cli/scaffolding) |
| `harness.dev app <name>` | Scaffold a new app | [Scaffolding](/cli/scaffolding) |
| `harness.dev publishers register --handle <handle>` | Claim a publisher handle | [Publishing](/cli/publishing) |
| `harness.dev publishers me` | Show your publisher record | [Publishing](/cli/publishing) |
| `harness.dev publish` | Submit an app for review + signing | [Publishing](/cli/publishing) |
| `harness.dev publish status <id>` | Poll review state of a submission | [Publishing](/cli/publishing) |
| `harness.dev install <publisher>/<short>[@semver]` | Install a signed app into the current project | [Installing](/cli/installing) |

## Authentication

Commands that hit `api.lloyal.ai` (anything under `publishers`, `publish`, plus pre-release `install` flows) are gated by Cloudflare Access. Two modes:

**Interactive (developer terminals).** The first authenticated command from a fresh terminal opens your browser to Cloudflare Access SSO (RFC 8252 OAuth + PKCE + dynamic client registration). Complete the one-time email PIN and the CLI caches tokens at `~/.cache/lloyal/auth.json` (mode 0600), refreshing transparently on each invocation — you see the browser only on first run and after the refresh token expires. To reset: `rm ~/.cache/lloyal/auth.json`.

**Service Token (CI).** Set both env vars and the CLI skips the browser flow entirely:

```bash
export CF_ACCESS_CLIENT_ID=<client-id>.access
export CF_ACCESS_CLIENT_SECRET=<client-secret>
```

Service Tokens are mapped server-side to a publisher account — CI publishes under the same handle as the team that owns it. **Service Tokens cannot register publisher accounts**: run `harness.dev publishers register` from an interactive terminal once; CI can then publish under that record.

**What's NOT authenticated:** catalog, manifest, and tarball fetches from `apps.lloyal.ai` are public read — all bytes are Ed25519-verified against the framework-vendored trust root. And plain `npm ci` against a committed lockfile needs no `harness.dev` at all.

**Troubleshooting:** `401 invalid_token` → stale token; any command triggers a refresh. `403 unauthorized` → your Access identity doesn't match an allow policy, or the Service Token is invalid (`DEBUG=harness.dev:*` shows the headers sent). `403 no-publisher-account` → register a handle first ([Publishing](/cli/publishing)).

## Typical workflows

**As an app developer:**

```bash
# One-time setup
harness.dev publishers register --handle acme

# Scaffold a new app
harness.dev app jira
cd jira
# ... write your app ...

# Publish it
harness.dev publish
harness.dev publish status <submission-id>   # poll until approved
```

**As a harness developer (consumer of apps):**

```bash
# Scaffold a new harness
harness.dev my-harness
cd my-harness

# Install signed apps from the channel
harness.dev install lloyal/web
harness.dev install acme/jira

# Then in your harness code:
# import { createWebApp } from '@lloyal-labs/web-app';
# import { createJiraApp } from '@acme/jira-app';
```

## How signing + verification works

Every app on `apps.lloyal.ai` is Ed25519-signed by the Lloyal platform key (`lloyal-platform-2026-q2`). The trust root is **vendored in `@lloyal-labs/rig`** — `harness.dev install` verifies signatures against the bundled public key. You don't trust the catalog server; you trust the key bytes shipped in the SDK you `npm install`'d.

The signed catalog at `apps.lloyal.ai/v1/catalog.json` lists every approved app's name, version, signed manifest URL, and tarball URL. Installs fetch the catalog (signature-verified), resolve the version, fetch the manifest (signature-verified), then fetch and signature-verify the tarball bytes before running `npm install` against the canonical URL.

The lockfile records the canonical channel URL + sha512 integrity, so subsequent `npm ci` reproduces the install with **no `harness.dev` required in CI**.
