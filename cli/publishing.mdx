---
title: "Publishing apps"
description: "`harness.dev publish` — submit an HDK app to apps.lloyal.ai for review and signing."
---

When your app is ready to ship, `harness.dev publish` packages it via `npm pack` and submits the tarball to the channel for review. Every submission enters a quarantine queue; on reviewer approval, Lloyal Labs signs the tarball bytes with the platform Ed25519 key and writes the signed artifact + manifest + updated catalog to `apps.lloyal.ai`.

## Step 0 (once): register a publisher handle

Your handle identifies your apps in the catalog as `<handle>/<app-name>`. Registration is one-time per identity, requires an interactive terminal (Service Tokens can't register), and attests the [Publisher ToS](/licensing/publisher-tos):

```bash
npx harness.dev publishers register --handle acme [--yes]

# Publisher ToS (v1): https://docs.lloyal.ai/licensing/publisher-tos
# Accept the publisher ToS? [y/N] y
# registered as acme
```

`<handle>` matches `[a-z][a-z0-9_-]{1,63}`. Only `lloyal` is reserved; everything else is first-come-first-served, subject to good-faith use. Re-running with the same handle is a no-op; a *different* handle returns `409 already-registered` (ownership transfers go through `support@lloyal.ai`). Registration errors:

| Error | Cause / fix |
|---|---|
| `handle-invalid` | Doesn't match the grammar — lowercase ASCII, 2–64 chars, `_`/`-` only |
| `handle-taken` / `handle-reserved` | Pick a different handle |
| `tos-required` | Re-run and accept the ToS (or pass `--yes`) |
| `already-registered` | Your identity has a handle — see `harness.dev publishers me` |
| `service-tokens-cannot-register` | Register interactively first; CI then publishes under that record |

Inspect your record anytime with `npx harness.dev publishers me` — handle, `status` (`active`/`suspended`), approved-version count, and the `tosVersion` you attested (a `tos-version-stale` error on publish means re-attest via `publishers register --yes`). The ToS in one breath: you retain ownership and license Lloyal to inspect/sign/distribute approved tarballs; you submit only non-malicious, protocol-compliant code; every submission is reviewed, outcomes communicated, rejected artifacts kept 30 days for appeal; material ToS revisions require re-attestation.

## Prerequisites per app

- Your app's `package.json` `name` is the npm import name (e.g., `@acme/jira-app`) — the symbol consumers `import` from once installed.
- Your `package.json` `version` is the catalog version. The catalog publishes your app as `<your-handle>/<short-name>@<version>`.
- Your `app.json` `name` is the short name (e.g., `jira`). If you prefix it with your handle (`acme/jira`), the prefix must match your account — otherwise the CLI auto-prefixes.

## Submit for review

```bash
npx harness.dev publish [--dir <app-directory>]
```

Defaults to `process.cwd()`. The CLI:

1. Looks up your publisher handle via `GET /v1/publishers/me`.
2. Validates your `app.json` and `package.json` shape.
3. Auto-prefixes the catalog `name` as `<your-handle>/<app.json.name>`.
4. Runs `npm pack` in the app directory to produce a tarball.
5. POSTs the tarball + a manifest stub to `https://api.lloyal.ai/v1/publish`.
6. The server quarantines the submission (writes pending bytes to R2, records a `pending` submission in KV) and returns a `submissionId`.

**Example:**

```bash
$ npx harness.dev publish --dir packages/apps/web
# harness.dev publish: submitting lloyal/web@1.0.0 (29724 bytes) to https://api.lloyal.ai/v1/publish...
# submitted lloyal/web@1.0.0 for review
#   submission: c5eb4751-019d-4930-a59d-1eff795de7f6
#   status:     pending
#   import:     @lloyal-labs/web-app
#   submitted:  2026-06-08T05:37:43.823Z
#   poll:       harness.dev publish status c5eb4751-019d-4930-a59d-1eff795de7f6
```

Save the `submissionId` — you use it to poll the review state.

## Poll review state

```bash
npx harness.dev publish status <submissionId>
```

Returns the current review state for a submission you own. Other publishers' submissions are not visible.

**While pending:**

```
c5eb4751-019d-4930-a59d-1eff795de7f6
  status:    pending
  submitted: 2026-06-08T05:37:43.823Z
```

**After approval:**

```
c5eb4751-019d-4930-a59d-1eff795de7f6
  status:    approved
  submitted: 2026-06-08T05:37:43.823Z
  approved:  2026-06-08T05:38:52.603Z
  reviewer:  review@lloyal.ai
  tarball:   https://apps.lloyal.ai/v1/bundles/lloyal__web-1.0.0.tgz
```

The signed tarball is now live in the catalog. Anyone can `npx harness.dev install lloyal/web` against it.

**After rejection:**

```
c5eb4751-019d-4930-a59d-1eff795de7f6
  status:    rejected
  submitted: 2026-06-08T05:37:43.823Z
  rejected:  2026-06-08T05:38:52.603Z
  reviewer:  review@lloyal.ai
  reason:    Manifest claims tools=[search,fetch] but tarball ships only [search].
             Resubmit with consistent declarations.
```

Address the reviewer's feedback and resubmit (a fresh `harness.dev publish` produces a new `submissionId`). Rejected pending artifacts are retained for 30 days as an appeals window.

## What the server validates

The Worker performs these checks before quarantining a submission. Failures return `400` with a structured error code:

| Error code | Cause |
|---|---|
| `no-publisher-account` | Your identity has no publisher record. Run [`publishers register`](/cli/publishing) first. |
| `account-suspended` | Your publisher account is suspended. Email `support@lloyal.ai`. |
| `tos-version-stale` | Your `tosVersion` is older than the current ToS. Re-attest via `publishers register --yes`. |
| `rate-limit-exceeded` | You've hit the per-identity rate limit (currently 10 publishes / 24h). Response includes `Retry-After`. |
| `invalid-name` | The catalog name doesn't match `<publisher-handle>/<app-name>`. |
| `name-handle-mismatch` | The handle prefix in `app.json.name` doesn't match your publisher record. |
| `invalid-version` | `package.json.version` is not a valid semver. |
| `size-mismatch` | The tarball's actual byte size doesn't match the manifest stub's claim (very rare; usually a packaging bug). |
| `importName-required` | `package.json.name` is missing. |
| `importName-mismatch` | The `package.json.name` inside the tarball doesn't match the manifest stub's `importName` (someone is lying — refuse). |
| `version-package-json-mismatch` | The `package.json.version` inside the tarball doesn't match the manifest stub's `version` (same — refuse). |
| `tarball-inspect-failed` | The Worker couldn't extract `package/package.json` from the tarball (invalid gzip, not a valid npm-pack output, etc.). |

## What happens on approval

On reviewer approval, the Worker:

1. Reads the pending tarball bytes from R2.
2. Re-extracts `package.json` defensively (same validation as at submit time).
3. Ed25519-signs the tarball bytes with the platform key.
4. Computes the npm-compatible `sha512-...` integrity.
5. Builds the final signed manifest with the signature, integrity, your publisher key id, and the tarball's `peerDependencies`.
6. Writes the canonical paths: `apps.lloyal.ai/v1/bundles/<handle>__<short>-<version>.tgz` + corresponding `.manifest.json`.
7. Reads the current catalog, inserts your version entry, re-signs the catalog, writes it back.
8. Updates the submission record to `approved`, bumps your publisher's `publishedCount`.
9. Deletes the pending R2 artifacts.

From the moment the catalog write completes, your app is publicly installable.

## What does NOT change

Tarball bytes are signed exactly as you submitted them. Lloyal Labs does not modify your code. If a revision is needed, you submit a new version with a new semver.

The signature is over the tarball bytes — anyone can verify integrity by fetching the tarball + manifest from `apps.lloyal.ai` and Ed25519-verifying against the framework-vendored `LLOYAL_PLATFORM_KEY_2026_Q2` public key.

## Rate limits

| Limit | Window | Per |
|---|---|---|
| 10 publishes | 24 hours | Per authenticated identity |
| 1 publisher registration | (forever) | Per email |

Rate limits are tracked in Workers KV with a sliding-window counter. Exceeding returns `429` with a `Retry-After` header.
