---
title: "Installing AgentApps"
description: "`harness.dev install` — fetch, signature-verify, and install signed AgentApps from apps.lloyal.ai into your harness."
---

`harness.dev install` is how a harness consumer adds a signed AgentApp to their project. It fetches the catalog and the requested tarball from `apps.lloyal.ai`, signature-verifies every step against the framework-vendored trust root, and runs `npm install` against the canonical channel URL so the consumer's `package-lock.json` records the canonical URL + sha512 integrity.

After the initial install, plain `npm ci` reproduces the deps with **no `harness.dev` involvement** — CI doesn't need to know about the channel.

## Install a signed AgentApp

```bash
npx harness.dev install <publisher>/<short>[@<semver>] [--allow-scripts]
```

**Examples:**

```bash
# Install the latest lloyal/web
npx harness.dev install lloyal/web

# Install at a specific version
npx harness.dev install lloyal/web@1.0.0

# Install with a semver range (highest matching is picked)
npx harness.dev install acme/jira@^2.0.0
```

The CLI:

1. Fetches `apps.lloyal.ai/v1/catalog.json`; Ed25519-verifies its signature against `CHANNEL_TRUST_ROOTS` (vendored in `@lloyal-labs/rig`).
2. Resolves `<publisher>/<short>[@<semver>]` to a specific catalog version entry. The entry carries `importName` — the npm package name to install.
3. Fetches the entry's `manifestUrl`; signature-verifies it; cross-checks `name` / `version` / `sizeBytes` against the catalog entry.
4. Fetches the tarball at `entry.tarballUrl`; Ed25519-verifies against the manifest's signature.
5. Computes the sha512 integrity locally for the post-install lockfile audit.
6. Caches the verified tarball at `$XDG_CACHE_HOME/lloyal/apps/<publisher>__<short>-<version>.tgz`.
7. Seeds npm's HTTP cache so step 8 resolves from local cache (no second download).
8. Runs `npm install --ignore-scripts --save <canonical-tarball-URL>` in `process.cwd()`. The lockfile records the canonical channel URL + sha512.
9. Re-reads the lockfile and confirms npm-recorded integrity matches what we computed pre-install. If they diverge, the package is uninstalled and the command errors (defense against an MITM between our verify-fetch and npm's install-fetch).

**Successful output:**

```bash
$ npx harness.dev install lloyal/web
# harness.dev install: resolved lloyal/web → lloyal/web@1.0.0
# harness.dev install: running `npm install --ignore-scripts --save https://apps.lloyal.ai/v1/bundles/lloyal__web-1.0.0.tgz` in /Users/dev/my-harness...
# (npm output)
# installed lloyal/web@1.0.0
#   package:   @lloyal-labs/web-app
#   tarball:   https://apps.lloyal.ai/v1/bundles/lloyal__web-1.0.0.tgz
#   integrity: sha512-GpJXpc+qHA6SI...
#   cached:    /Users/dev/.cache/lloyal/apps/lloyal__web-1.0.0.tgz
```

## After install

```ts
// In your harness code:
import { createWebApp } from '@lloyal-labs/web-app';
import { createAppRegistry } from '@lloyal-labs/rig';

const registry = yield* createAppRegistry({
  apps: { web: createWebApp },
});
```

The `importName` field in the catalog (`@lloyal-labs/web-app` in this case) is the npm package name — the catalog name `lloyal/web` is the channel-side identifier; the harness code imports by the npm package name.

## CI reproducibility

After your first `harness.dev install`, the resulting `package.json` and `package-lock.json` look like:

```json
// package.json
{
  "dependencies": {
    "@lloyal-labs/web-app": "https://apps.lloyal.ai/v1/bundles/lloyal__web-1.0.0.tgz"
  }
}
```

```json
// package-lock.json (excerpt)
{
  "packages": {
    "node_modules/@lloyal-labs/web-app": {
      "version": "1.0.0",
      "resolved": "https://apps.lloyal.ai/v1/bundles/lloyal__web-1.0.0.tgz",
      "integrity": "sha512-GpJXpc+qHA6SI..."
    }
  }
}
```

Commit both files. In CI:

```bash
npm ci
```

That's it. npm fetches the canonical URL, verifies the sha512 integrity (matches what we signed), extracts. **No `harness.dev` step in CI.**

This is the key portability win: every harness that committed a lockfile after `harness.dev install` can deploy through any standard Node.js pipeline — Docker images, GitHub Actions, Vercel, AWS Lambda — without leaking the channel-aware tooling into the deploy path.

## Cache

`harness.dev install` writes a verified-tarball cache at:

```
$XDG_CACHE_HOME/lloyal/apps/<publisher>__<short>-<version>.tgz
```

(Falls back to `~/.cache/lloyal/apps/` if `$XDG_CACHE_HOME` is unset, which matches the XDG default.)

If the cache hits on a subsequent install, `npm install` resolves from disk — no second download.

To clear the cache:

```bash
rm -rf ~/.cache/lloyal/apps/
```

## --allow-scripts

By default the CLI passes `--ignore-scripts` to npm, suppressing any `preinstall` / `postinstall` / `install` hooks the package declares. The Ed25519 signature attests to the **bytes' provenance** (this came from Lloyal Labs's signing key after their review) — not to the safety of arbitrary install-time code execution.

If you trust the specific publisher's install scripts (e.g., a native-build step you've audited):

```bash
npx harness.dev install --allow-scripts <publisher>/<short>
```

Most AgentApps don't need install scripts. The default is safe-by-default.

## Error cases

| Error | Cause |
|---|---|
| `AppNotFoundError: App "..." is not listed in the catalog` | The publisher hasn't shipped this AgentApp yet, or the name is wrong. |
| `AppNotFoundError: App "..." has no version matching "^X.Y.Z"` | The semver range matches none of the published versions. The error lists what versions exist. |
| `BundleVerificationError: Catalog at ... failed Ed25519 signature verification` | Either the catalog is corrupted (rare — file an issue) or the framework-vendored trust root is outdated (update `@lloyal-labs/rig`). |
| `BundleVerificationError: Tarball size N does not match manifest.sizeBytes M` | The catalog and tarball are out of sync. File an issue. |
| `BundleVerificationError: Ed25519 signature verification failed for ...@...` | The tarball bytes don't sign-verify. **Do not install.** File an issue. |
| `integrity mismatch — npm-installed bytes did not match our pre-verified bytes` | An MITM swapped bytes between our verify-fetch and npm's install-fetch. The package is automatically uninstalled. |

The `BundleVerificationError` messages are intentionally noisy — they're the trust-boundary signal. Don't write code that swallows them.
