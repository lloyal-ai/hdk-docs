AgentApps run **in-process** with the model. That's the perf story ŌĆö no IPC, no serialization on the hot tool path, shared spine across hundreds of agents ŌĆö and it's also the security story: when there's no process boundary, the trust boundary becomes *the prompt itself*. Everything on this page exists to keep that boundary load-bearing.

## The threat model

Three concrete attacks the AgentApp protocol is designed against:

1. **Catalog-block escape.** An AgentApp-supplied string (manifest `name`, `useWhen`, etc.) interpolated into the shared spine could include a newline, a fake chat-role marker (`SYSTEM:`), or a markdown code fence ŌĆö convincing the model that the catalog is over and the prompt is now somewhere else.
2. **Cross-AgentApp prompt injection.** AgentApps share one prompt surface. A web page fetched by `web_research` could contain text instructing the model to call a `corpus_research` tool or a sensitive write tool, riding the shared spine into a different AgentApp's discipline.
3. **Unauthorized writes.** A model that decides to call a write tool ŌĆö because of a planner mistake, a prompt-injected page, or a confused recovery prompt ŌĆö must not be able to actually perform the write without the *session* (not the model) having consented.

The defense stack mitigates all three with primitives the AgentApp protocol makes structural.

## Defense 1 ŌĆö metadata sanitization at `defineApp`

AgentApp-supplied strings that enter the shared prompt are grammar-constrained at `defineApp` time. The checks fire synchronously, throw on failure, and the AgentApp can't enter the registry:

- **`name` / `protocol.name`** match `[a-z][a-z0-9_-]{1,63}`. Lowercase ASCII start, alphanumeric + `_` / `-`, 2-64 chars. No `*` (would break the markdown bold in the boundary marker), no whitespace, no control chars.
- **`protocol.useWhen`** is bounded length (max 280 chars), no chat-role markers (`SYSTEM:`, `USER:`, `ASSISTANT:`), no markdown code fences (` ``` `), no `\n` or `\r`.
- **`skill`** (when string-typed) MUST NOT contain the literal `Apply the **` substring ŌĆö the framework prepends `BOUNDARY_MARKER(protocol.name)`, so an `skill.eta` carrying that line would emit the boundary marker twice.
- **`protocol.tools[]`** is a non-empty unique array of name-regex strings, and the tool map's keys must equal it as a set.

These rules close the catalog-escape vector. An attacker who controls an AgentApp's manifest can't break out of the catalog block; the worst case is a rejected AgentApp that never reaches the spine. The constants and validations live in `@lloyal-labs/rig/src/protocol.ts` and `define-app.ts`.

## Defense 2 ŌĆö per-spawn prose isolation

The framework prefixes every per-spawn user message with `BOUNDARY_MARKER(protocol.name)`:

```
Apply the **corpus_research** protocol.

<the rendered skill template for this spawn>
```

This tells the model *which AgentApp's discipline applies to this turn*. A spawn for `corpus_research` reads its task under the corpus protocol's discipline; a spawn for `web_research` under the web protocol's. Combined with the `TOOL_SELECTION_RULE` at the spine root (*"use only that protocol's tools"*), the model has both a global routing rule and a per-turn assignment.

This isn't a hard isolation ŌĆö the model could still call any tool the catalog lists (the grammar permits the full set, for KV-amortization reasons). What it provides is **routing discipline** that any tool dispatch outside the assigned AgentApp becomes a candidate for refusal.

## Defense 3 ŌĆö `protected` + `authGuard` + grants

The real hard gate for *consequential* actions lives at DISPATCH, not in the prompt.

### Marking a tool protected

```ts
class SendEmailTool extends Tool<{ to: string; subject: string; body: string }> {
  readonly name = "send_email";
  readonly protected = true;     // ŌåÉ write/sensitive
  // ...
}
```

`protected: true` declares *"this tool needs an explicit grant before it runs."* The framework's authGuard reads the calling agent's lineage at dispatch time and asks the session's `GrantStore`: does the session hold a grant for `send_email`?

### The grant store

```ts
export interface GrantStore {
  has(toolName: string): Operation<boolean>;
  grant(toolName: string): Operation<void>;
  revoke(toolName: string): Operation<void>;
  granted(): Operation<readonly string[]>;
}
```

A `GrantStore` lives on `GrantStoreCtx`, set by the harness at boot. Semantics from `agents/src/grant-store.ts`:

- **Binary per tool.** A grant is keyed by tool name; either the session holds it or it doesn't. No scopes, no expiry in the base protocol ŌĆö richer policies are a harness concern.
- **Fail-closed.** With no grant store on `GrantStoreCtx`, no grants exist: every protected tool is denied. Open (non-protected) tools are unaffected.
- **Acquired via consent**, never via the model. The model triggers the call; the runtime holds the grant; the tool's `execute` uses the underlying secret. The credential never enters the model's context.

### How dispatch fails closed

When the model emits a tool call for a protected tool the session hasn't granted:

1. The pool's DISPATCH phase sees `tool.protected === true`.
2. `authGuard` reads `GrantStoreCtx.expect()` and calls `has("send_email")`.
3. `has` returns `false` ŌåÆ dispatch denies the call.
4. The pool emits `tool:authReject` (carrying the agent's `walkAncestors` lineage so the trace records *which* agent attempted *which* tool from *which* lineage), then *nudges* the agent ŌĆö the agent sees an error string at the call position and can recover (try a different tool, report what's missing).

The model never executes the action. The session never produces the side effect. The user is never billed for an email they didn't authorize.

### Acquiring grants

**Harness consent prompts.** The harness asks the user before calling `grantStore.grant("send_email")` ŌĆö a CLI tool prompts to confirm; a desktop app shows a system dialog. Consent moves from user ŌåÆ grant store; the model triggers, the runtime executes.

### Collecting credentials

An AgentApp that needs a credential (an API key, an OAuth token ŌĆö `web_research` against a paid search backend, a Gmail AgentApp) declares a **`configSchema`** in its manifest. The app never collects the secret itself. The harness gathers it out-of-band ŌĆö a setup prompt, an env var, a `scripts/connect.ts` helper for an OAuth handoff ŌĆö and writes it with **`AppConfigStore.set(appName, config)`**.

On `registry.enable(...)` the stored config is validated against the app's `configSchema`; at run time the app reads it via the **`AppConfigStoreCtx`** context ŌĆö `yield* (yield* AppConfigStoreCtx.expect()).get(appName)` (an `Operation` ŌĆö `yield*` it). Tools that touch the credential are marked `protected`, so they stay behind the `authGuard` grant until the user approves. Credential moves from user ŌåÆ config store; consent moves from user ŌåÆ grant store.

## Why this works in-process

In an out-of-process model (MCP-style), the trust boundary is the process ŌĆö a compromised server is sandboxed by the OS. In-process AgentApps can't rely on that, so the AgentApp protocol trades isolation for:

- **Structural sanitization** of every AgentApp-supplied string that hits the prompt (Defense 1).
- **Bytes-locked AgentApp protocol** that the model is trained-on/aware-of, narrowing what the model treats as authoritative (Defense 2 ŌĆö the `BOUNDARY_MARKER` + `TOOL_SELECTION_RULE` + per-spawn discipline).
- **Hard dispatch-time gating** for any tool that takes a consequential action (Defense 3 ŌĆö `protected` + `authGuard` + grants).

The net is: an attacker who controls an AgentApp's metadata can't escape the catalog; a prompt-injected page can't trigger an unauthorized write; a confused recovery prompt can't either. The residual risk is that an attacker convinces *the user* to grant a protected tool ŌĆö same residual as any consent-driven system.

## Residual risk: findings as trusted text

There is one important residual that AgentApp authors and harness authors should both know:

**Tool results that pass through synth are currently treated as trusted prose.** A `fetch_page` retrieves a web page; the agent reports a finding; synth reads the finding and writes a report. The page's text can ride the finding into synth and influence the report. If synth has tools (some harnesses give it search/fetch for verification), a malicious finding could steer those calls.

Mitigations available today:

- **Synth without tools.** Pass no tools to the synth `useAgent`; it can only synthesize prose, not execute actions. (Trivial fix; the reasoning.run harness does this.)
- **Per-spawn prose isolation extends to findings.** Each finding renders inside its boundary marker, so synth knows it came from a specific AgentApp's protocol.

The rule that follows: if you're building an AgentApp with write capabilities, do **not** rely on findings-prose-as-trusted; gate every consequential action behind `protected`.

## Author checklist

When you author an AgentApp that exposes writes or sensitive reads:

- **Mark protected anything that mutates external state or could exfiltrate.** Writes, deletes, sends, transfers ŌĆö and also fetches of arbitrary URLs the user didn't supply (an exfiltration surface even if it looks like a read).
- **Keep credentials out of the model's prompt.** Collect them via `configSchema` + `AppConfigStore`, read them via `AppConfigStoreCtx` in the tool's `execute` body. The model only triggers.
- **Validate `args` in `execute`.** Just because the grammar made `args` structurally valid doesn't mean it's semantically safe ŌĆö sanitize URLs, bound payload sizes, refuse out-of-scope targets.
- **Surface honest errors.** Return `{ error: "..." }` for denied/invalid calls rather than throwing ŌĆö the agent gets a chance to recover with a different approach.
- **Test the authGuard path.** Two layers worth asserting. First, that your tool is actually marked protected ŌĆö `expect(mySendTool.protected).toBe(true)`. Second, the grant gate itself, driven directly against a `GrantStore` (its methods are all Effection `Operation`s ŌĆö `yield*` them):

```typescript
import { run } from "effection";
import { createGrantStore } from "@lloyal-labs/rig";

await run(function* () {
  const grants = createGrantStore();
  // fail-closed: no grant ŌåÆ denied
  expect(yield* grants.has("send_email")).toBe(false);
  // after the harness records consent ŌåÆ allowed
  yield* grants.grant("send_email");
  expect(yield* grants.has("send_email")).toBe(true);
});
```

The framework's `authGuard` consults exactly this at DISPATCH: a protected tool with no grant is denied and emits `tool:authReject` before `execute()` ever runs.
