**A `Tool` is anything an agent can call** — a JSON-schema-described function with an Effection-generator body. Every AgentApp declares its tools in the manifest's `protocol.tools[]` and supplies matching `Tool` instances via `defineApp`.

## The base class

```ts
export abstract class Tool<TArgs = Record<string, unknown>> {
  /** Tool name — the function identifier in tool calls. */
  abstract readonly name: string;

  /** Human-readable description shown to the model. */
  abstract readonly description: string;

  /** JSON Schema describing the tool's expected arguments. */
  abstract readonly parameters: JsonSchema;

  /** Whether invoking this tool requires a session-held grant. */
  readonly protected?: boolean;

  /** Execute with parsed arguments. Returns an Effection Operation. */
  abstract execute(args: TArgs, context?: ToolContext): Operation<unknown>;

  /** Optional reasoning probe prefilled after this tool's result. */
  probe(_result: unknown): string | null { return null; }

  /** Auto-generated OpenAI-compatible function schema. */
  get schema(): ToolSchema { /* built from name + description + parameters */ }
}
```

You never construct the schema manually — the getter derives it. Pass instances straight to `agent` / `agentPool` (`tools: [new SearchTool(...), reportTool]`); the framework serializes schemas for the prompt and builds the dispatch map internally.

## Walkthrough: a search tool

The corpus AgentApp's `SearchTool` shows every concern:

```ts
import { call } from "effection";
import { Tool, Trace } from "@lloyal-labs/lloyal-agents";
import type { JsonSchema, ToolContext } from "@lloyal-labs/lloyal-agents";

export class SearchTool extends Tool<{ query: string }> {
  readonly name = "search";
  readonly protected = false;
  readonly description =
    "Search the knowledge base. Returns sections ranked by relevance with line ranges for read_file.";

  readonly parameters: JsonSchema = {
    type: "object",
    properties: { query: { type: "string", description: "Search query" } },
    required: ["query"],
  };

  constructor(private chunks: Chunk[], private reranker: Reranker, opts?: { threshold?: number }) {
    super();
    this._threshold = opts?.threshold ?? 0;
  }

  *execute(args: { query: string }, ctx?: ToolContext): Operation<unknown> {
    const query = args.query?.trim();
    if (!query) return { error: "query must not be empty" };

    const results = yield* call(async () => {
      let last: ScoredChunk[] = [];
      for await (const { results, filled, total } of this.reranker.score(query, this.chunks)) {
        ctx?.onProgress?.({ filled, total });
        last = results;
      }
      return last;
    });

    const passing = results.filter((r) => r.score >= this._threshold);
    const topRejected = passing.length === 0 ? results.slice(0, 3) : [];
    return { hits: passing, thresholdScore: this._threshold, totalScored: results.length, topRejected };
  }
}
```

The patterns to copy:

- **Args via type parameter.** `Tool<{ query: string }>` types `execute`'s first argument.
- **`name` matches the manifest.** Must exactly equal an entry in `protocol.tools[]`; `defineApp` validates the set match and that each instance's `.name` equals its map key.
- **`description` is what the model reads** to pick between tools and plan follow-ups. Be precise about what's returned (line ranges, score scales).
- **JSON Schema → GBNF.** `parameters` compiles to a sampler-level grammar — the model can only emit tokens that produce structurally valid arguments. Invalid JSON is unreachable, not just rare.
- **`*execute` is an Effection generator.** Async work goes through `yield* call(() => promise)` — never `await` inside the generator. Inside you have `Trace`, `Ctx`, `Store`, and you can spawn sub-agents.
- **Constructor = dependency injection.** Tools are instantiated before the pool runs (typically in the AgentApp factory) — corpora, rerankers, API clients arrive via the constructor, never via globals.
- **Return value is JSON-serialized** and prefilled into the agent's KV at the call position; the next PRODUCE attends over it.

## Errors: return, don't throw

A thrown error terminates the agent (`tool_error`); a **returned** error object is prefilled as a tool result the agent can adapt to:

```ts
*execute(args: { pattern: string }): Operation<unknown> {
  if (!args.pattern?.trim()) return { error: "pattern must not be empty" };
  try { new RegExp(args.pattern, "gi"); }
  catch { return { error: `Invalid regex: ${args.pattern}` }; }
  // ...
}
```

Zero-result responses should carry guidance — it changes the model's next move:

```ts
if (matches.length === 0) {
  return { totalMatches: 0, matches: [],
    note: "Zero matches. Try search() for semantic matching or a broader regex." };
}
```

The one deliberate throw: `ToolRetryError` for **transient** backend failures (rate limits). The pool parks the agent at zero KV cost and re-executes the same call after the delay — the model never sees infrastructure weather. See [Pressure & policy](/under-the-hood/kv-pressure#6-dispatch--exploreexploit-and-retry-parking).

## The `protected` bit

**Read tools (the default)** are open: any agent in the pool may call them — retrieval, file reads, web fetches; anything that *gathers* rather than mutates.

**Protected tools (`protected: true`)** are gated at dispatch by `authGuard` against a session-held grant. The model sees the tool and can emit a call; without a grant, the framework denies it and emits `tool:authReject`:

```ts
export class SendEmailTool extends Tool<{ to: string; subject: string; body: string }> {
  readonly name = "send_email";
  readonly protected = true;            // ← write/sensitive action
  readonly description = "Send an email via the user's connected account.";
  readonly parameters = /* ... */;

  *execute(args, ctx): Operation<unknown> {
    // Runs only if the session holds a grant for "send_email".
    return yield* call(() => this.smtp.send(args));
  }
}
```

The credential never enters the model's context: the session holds the grant, the tool body holds the secret, the model only triggers. If unsure whether to protect, protect — a harness can grant freely; an open write tool is much harder to recall. [AgentApp security](/build-an-app/app-security) covers the threat model and grant flows.

<Note>
Open-by-default isn't a permission downgrade — routing is already gated at the spine layer (the catalog lists only enabled AgentApps' tools, and every spawn is prefixed with `Apply the **<protocol>** protocol.`). `protected` is the orthogonal gate for *consequential* actions.
</Note>

## `ToolContext`

`execute()`'s second argument carries dispatch-time state:

```ts
interface ToolContext {
  agentId: number;
  branch?: Branch;             // calling agent's branch — for warm-forking sub-agents
  onProgress?: (p: { filled: number; total: number }) => void;
  scorer?: EntailmentScorer;   // entailment scorer, propagated to all depths
  explore?: boolean;           // true = explore mode, false = exploit (pool decides)
  pressurePercentAvailable?: number;
  peerHistory?: ToolHistoryEntry[];  // sibling agents' tool histories (excluding self)
}
```

**Cross-agent dedup via `peerHistory`.** The web AgentApp's `web_search` and `fetch_page` reject calls a sibling already made:

```ts
*execute(args: { query: string }, context?: ToolContext): Operation<unknown> {
  const queryLower = args.query.toLowerCase();
  if (context?.peerHistory?.some((h) => {
    if (h.name !== "web_search") return false;
    try { return (JSON.parse(h.args) as { query?: string }).query?.toLowerCase() === queryLower; }
    catch { return false; }
  })) {
    return { error: "Resource unavailable. Try a different query." };
  }
  // ... normal execution
}
```

The agent reads "Resource unavailable" like any failed call and diversifies — no special handling needed. (Same-agent dedup is the policy's job via lineage-aware tool guards.)

**Progress** surfaces as `agent:tool_progress` events for TUI display — call `ctx.onProgress({ filled, total })` inside long operations.

## Recursive tools — the AgentApp author's deep-agents surface

`execute()` returns an `Operation`, so a tool can spawn an entire sub-pool. Pass `context.branch` as `parent` and the sub-agents fork the **calling agent's live attention** — system prompt, tool schemas, every prior result — for the cost of a turn separator:

```ts
*execute(args: { questions: string[] }, context?: ToolContext): Operation<unknown> {
  const pool = yield* agentPool({
    parent: context?.branch,           // warm fork — Continuous Context
    tools: [searchTool, reportTool],
    terminal: reportTool,
    orchestrate: parallel(
      args.questions.map((q) => ({ content: q, systemPrompt: SUB_PROMPT })),
    ),
  });
  return { results: pool.agents.map((a) => a.result).filter(Boolean) };
}
```

Returning `{ results: string[] }` (or `{ nestedResults: string[] }`) makes the findings bubble up into the calling agent's `nestedResults` automatically. Depth is bounded by KV pressure, not a hardcoded limit.

Before writing a custom recursive tool, reach for **`DelegateTool`** from `@lloyal-labs/rig` — it's this pattern productized, plus the safeguards a hand-rolled version won't have: entailment gating on proposed sub-tasks and echo detection against the delegating agent's own task. Configuration and the gating behavior: [Agents — Deep agents](/build-a-harness/agents-and-pools#deep-agents).

## Terminal tools

A terminal tool ends an agent's turn. The pool intercepts the call at the policy layer and captures the `result` argument as the agent's return — **`execute()` never runs**:

```ts
class ReportTool extends Tool<{ result: string }> {
  readonly name = "report";
  readonly description = "Submit your final research findings";
  readonly parameters: JsonSchema = {
    type: "object",
    properties: { result: { type: "string", description: "Your findings and answer" } },
    required: ["result"],
  };
  *execute(): Operation<unknown> { return {}; }
}
```

Pass it **by reference** — `agentPool({ tools, terminal: reportTool })` — never by name string. The pool enforces work-before-report: when the registry contains non-terminal tools, a first-call report is nudged back to work; reporter sub-pools (terminal only) skip the check. `reportTool` from `@lloyal-labs/rig` is the stock instance; harnesses define their own when the turn should end in an action (`send_email`, `persist`).

## The probe hook

`probe(result): string | null` lets a tool nudge the model to **reason in prose** about a result before the next tool call. The pool prefills the probe text after the tool result, before the lazy grammar resets — so the next sampled tokens are reflection, not another call:

```ts
probe(result: unknown): string | null {
  if (typeof result === "object" && result && "hits" in result) {
    const r = result as { hits: unknown[]; topRejected: unknown[] };
    if (r.hits.length === 0 && r.topRejected.length > 0) {
      return "Reflect: nothing matched above the floor. Broaden the query, or look elsewhere?";
    }
  }
  return null;
}
```

The probe receives the result, so it can be conditional — return `null` to skip (zero cost). Probes fire only after real dispatches, never on nudges or terminal interceptions. Use sparingly: every probe extends the agent's KV. They earn their tokens on dense results (semantic search, large reads); they're noise on routine ones.

## How dispatch works

The contract you write to:

1. **The model emits a tool call**, grammar-constrained against your `parameters` — `args` is structurally valid when it lands.
2. **The pool calls `execute(args, ctx)`** inside a child scope of the tick fiber — cancellable, traceable, one tool at a time per pool.
3. **You return a value**; it's JSON-stringified and prefilled into the agent's KV at the call position.
4. **The next PRODUCE attends over the prefill** (plus your probe, if any), and sampling continues.

The whole loop is in-process — no IPC, no HTTP boundary. Tool order in the array matters: terminal tools should not be last, particularly next to a name-prefix sibling — [Grammar & ordering](/under-the-hood/grammar-and-ordering) has the recency-bias rule and the failure case.

## Authoring checklist

- **Type the args** (`Tool<{ query: string }>`), and validate edges in `execute` — return structured errors, don't throw.
- **Make `description` actionable** — the model plans multi-step sequences from it.
- **Inject dependencies via the constructor**; in production the AgentApp factory owns construction and the registry owns lifecycle ([What is an AgentApp](/build-an-app/what-is-an-app)).
- **Set `protected: true` on anything that mutates external state.**
- **Return small, structured payloads.** A 50 KB result is 50 KB the agent attends over for the rest of its life — trim at the tool boundary, not in synth.
- **Throw `ToolRetryError` for transient failures only**; return error objects for everything else.
- **Use `probe` sparingly.**
