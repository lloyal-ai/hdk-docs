title: Distribution
description: How AgentApps reach harnesses. The canonical channel is apps.lloyal.ai — a signed npm tarball installed into node_modules, framework-locked into @lloyal-labs/rig. First-party harnesses may also depend on AgentApps they own at build time.
---

An AgentApp is a **signed npm tarball**. There are **two ways its package reaches a harness's `node_modules`**, but they converge on the *same* runtime path — a plain static `import` of the factory, handed to `registry.enable`:

1. **The canonical channel** (`apps.lloyal.ai`) — for *any* AgentApp reaching a harness it doesn't own. Third-party AgentApps, first-party AgentApps published for external consumption — all flow through here, via `harness.dev install`.
2. **First-party build-time dependency** — for a harness vendor depending on an AgentApp **it owns** from a private source (private monorepo, private registry, git+ssh dep). An ordinary `npm` dependency, not a parallel public path.

The framework provides **no runtime "load app by name" verb.** Acquisition (getting the tarball into `node_modules`) is a build/install-time step; loading is a static `import` like any other dependency.

## The canonical channel — `apps.lloyal.ai`

Every AgentApp published to `apps.lloyal.ai` is reviewed by Lloyal Labs for Source-contract conformance, tool-safety review (which tools are `protected`, what data they touch, what consent they require), manifest validation, and signature provenance before listing. The catalog at `apps.lloyal.ai/v1/catalog.json` is the verified registry that consumers rely on.

The framework points at this verified registry by default. `harness.dev install` takes a name, not a URL; the catalog URL (`CHANNEL_CATALOG_URL`) and trust roots (`CHANNEL_TRUST_ROOTS`) are compile-time constants in `@lloyal-labs/rig/src/protocol.ts`. This way every harness built on HDK resolves AgentApps through the same verified channel by default — AgentApps interoperate across harnesses, consumers see one coherent ecosystem, and the AgentApp protocol does not fragment into incompatible sub-catalogs.

```
   AUTHOR                         PLATFORM                          CONSUMER
 ┌──────────────────┐        ┌─────────────────────┐        ┌──────────────────────┐
 │ harness.dev      │        │ api.lloyal.ai        │        │ harness.dev install  │
 │  publish         │        │  /v1/publish         │        │  lloyal/web          │
 │  ↓ npm pack .tgz │──POST─▶│  ↓ quarantine        │        │  ↓ Ed25519-verify    │
 │  + attention-    │  tgz + │  ↓ human review      │        │    catalog/manifest/ │
 │    surface.json  │manifest│  ↓ Ed25519-sign      │        │    tarball + sha512  │
 └──────────────────┘        │  ↓ write catalog.json│        │  ↓ npm install <url> │
                             └──────────┬──────────┘        │  ↓ node_modules/     │
                             signed catalog + tarball        │    @lloyal-labs/     │
                             at apps.lloyal.ai/v1/  ─────────▶│    web-app           │
                                                             │  ↓ auditLockfile     │
                                                             └──────────┬───────────┘
                                                        import { createWebApp }
                                                        yield* registry.enable(createWebApp)
```

### Installing — the harness operator flow

```bash
npx harness.dev install lloyal/web@^1.2.0   # resolve + verify + npm install
```

`harness.dev install`:

1. Fetches `CHANNEL_CATALOG_URL` (`https://apps.lloyal.ai/v1/catalog.json`) and Ed25519-verifies its signature against `CHANNEL_TRUST_ROOTS`. If verification fails the command errors — no trust path through a tampered catalog.
2. Resolves `lloyal/web@^1.2.0` to a specific catalog version entry (highest match, or `AppNotFoundError`). The entry carries `importName` — the npm package name to install.
3. Fetches the entry's manifest, verifies it, and cross-checks `name` / `version` / `sizeBytes` against the catalog entry.
4. Fetches the tarball at `entry.tarballUrl`, checks its byte size, and Ed25519-verifies the bytes against the manifest signature — the trust boundary.
5. Computes the sha512 integrity locally, then runs **`npm install --ignore-scripts --save <canonical-tarball-URL>`** in `process.cwd()`, so the package lands at `node_modules/<importName>/` like any ordinary dependency and the lockfile records the canonical URL + sha512.
6. **Audits the lockfile** (`auditLockfile`): npm's recorded `integrity`, `resolved` URL, and the `package.json` dependency spec must all match. On any mismatch the package is uninstalled and the command errors — defending against a swap between our verify-fetch and npm's install-fetch.

Every URL and every verification key is framework-vendored. After the first install, plain **`npm ci`** against the committed lockfile reproduces the dependency with **no `harness.dev` involvement** — CI doesn't know about the channel. See the [install reference](/cli/installing) for the cache, `--allow-scripts`, and error cases.

### Publishing — the AgentApp author flow

```bash
harness.dev publish   # auth → npm pack → submit to apps.lloyal.ai for review + signing
```

`harness.dev publish`:

1. Authenticates the publisher against `api.lloyal.ai` (interactive Cloudflare Access SSO from a developer terminal, or a Service Token in CI). Looks up the handle via `GET /v1/publishers/me` and scopes the catalog name as `<handle>/<app.json.name>`.
2. Generates `attention-surface.json` into the app directory — the signed disclosure of the app's `skill` prose, full tool schemas, `useWhen`, and `configSchema` — and runs **`npm pack`** to build the `.tgz`, verifying the attention surface actually landed inside the tarball.
3. POSTs the tarball + an unsigned manifest stub (`name`, `version`, `entry`, `importName`, `sizeBytes`, `peerDependencies`) to `https://api.lloyal.ai/v1/publish`. The submission enters a **quarantine review queue** and returns a `submissionId` to poll.

The platform side then, **on reviewer approval**:

4. Re-extracts and re-validates `package.json` from the pending tarball bytes.
5. Ed25519-signs the tarball bytes with the Lloyal platform key (`LLOYAL_PLATFORM_KEY_2026_Q2`, listed in `CHANNEL_TRUST_ROOTS`), computes the npm-compatible sha512 integrity, and writes the signed tarball + manifest to R2.
6. Reads `catalog.json`, inserts the version entry, re-signs the catalog with the platform key, and writes it back. From that moment the AgentApp is publicly installable.

There is no "self-hosted catalog" option, and no per-publisher signing keys: publishers authenticate to submit, but every signature on every byte in the catalog comes from the Lloyal platform key **after Lloyal's review**. The signature on a tarball is the cryptographic record of that review. Tarball bytes are signed exactly as submitted — Lloyal Labs never modifies your code; a revision is a new semver. See the [publish reference](/cli/publishing) for handle registration, the validation table, and review states.

### Loading — after install

Once the tarball is in `node_modules`, the harness loads it like any dependency — a static `import` of the factory, handed to the registry:

```ts
import { createWebApp } from "@lloyal-labs/web-app";   // the catalog entry's importName
import { createAppRegistry } from "@lloyal-labs/rig";

const registry = yield* createAppRegistry({ configStore });
yield* registry.enable(createWebApp);
```

The catalog name (`lloyal/web`) is the channel-side identifier; the harness code imports by the npm package name (`importName`, e.g. `@lloyal-labs/web-app`). The rig runtime primitives `resolveAppEntry`, `verifyBundle`, `CHANNEL_CATALOG_URL`, and `CHANNEL_TRUST_ROOTS` back the install CLI's verify path — the framework itself never evaluates app bundles at runtime; it imports installed packages. [Lifecycle & registry](/build-an-app/lifecycle-and-registry) covers `registry.enable` and the per-AgentApp scope model.

## First-party build-time dependency

A harness vendor MAY depend on AgentApps it owns at build time, from a private source. This is an ordinary `npm` dependency, **not** a public distribution path. The reasoning.run reference harness uses it for `@lloyal-labs/web-app` and `@lloyal-labs/corpus-app`:

```ts
import { createWebApp } from "@lloyal-labs/web-app";
import { createCorpusApp } from "@lloyal-labs/corpus-app";

const registry = yield* createAppRegistry({ configStore });
yield* registry.enable(createWebApp);
yield* registry.enable(createCorpusApp);
```

Constraints:

- The vendor must own the source repos (private monorepo, private registry, git+ssh with auth).
- Third-party AgentApps NEVER use this path. There is no "publish your AgentApp to public npm" story — third-party distribution goes through `apps.lloyal.ai` exclusively.
- Audience: a harness vendor shipping its own first-party AgentApps preinstalled.

If you don't own the AgentApp, you go through the canonical channel. Either way, the load path is identical: `import` the factory, `registry.enable` it.

A separate dev tool, distinct from the framework runtime, wraps both sides of the channel — scaffolding, publisher accounts, publish + install. **See the [CLI reference](/cli) for the full command surface and workflows.**
