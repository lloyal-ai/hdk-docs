**An AgentApp is what you publish so any HDK harness can load your capability — and this is all the model ever learns about it:**

```json app.json
{
  "name": "corpus",
  "appProtocolVersion": "3.0",
  "protocol": {
    "name": "corpus_research",
    "useWhen": "investigating a local document corpus — finding occurrences of terms, reading specific files at line offsets, semantic retrieval over indexed corpus content.",
    "tools": ["grep", "read_file", "search"]
  },
  "configSchema": {
    "type": "object",
    "required": ["corpusPath"],
    "properties": {
      "corpusPath": { "type": "string" }
    }
  }
}
```

That's the real manifest of the `corpus` reference AgentApp. The `protocol` block — a name, a `useWhen` routing sentence, a tool list — is what the registry decodes into the **shared spine** once, at boot; every forked agent inherits the whole catalog through attention and routes tasks by reading `useWhen` at execution time. Behind the manifest sit the implementations: a `Source` (data access + retrieval scoring), the `Tool`s named in `protocol.tools`, and a `skill` preamble rendered per spawn. `defineApp` wires the parts and refuses malformed ones before they can touch the prompt.

<Note>
AgentApps appear in code as `App` — `defineApp`, `AppManifest`, `app.json`. The docs say **AgentApp** to distinguish the published capability from the *application* you embed a harness in; the code keeps the short name.
</Note>

## Anatomy

`defineApp(spec): App` (in `@lloyal-labs/rig`) wires and validates five parts:

| part | type | role |
| --- | --- | --- |
| **manifest** | `AppManifest` (from `app.json`) | Declarative identity + the **protocol**: `name`, `protocol.name`, `protocol.useWhen`, `protocol.tools[]`, `appProtocolVersion`. |
| **source** | `Source` | Data access to a live system + retrieval intelligence: tools getter, `promptData()` (TOC/contents advert), `createScorer()` (reranker-backed `EntailmentScorer`). |
| **tools** | `Record<name, Tool>` | The actual actions. Each `Tool` has a JSON-schema `parameters`, an `execute()` generator, and a **`protected` flag** marking sensitive (write) ops. Keys must equal `protocol.tools[]` as a set — no missing, no extras. `defineApp` takes this **map**; the runtime exposes the resolved app's `tools` as an ordered **`readonly Tool[]`** (ordered by `protocol.tools`). |
| **skill** | `string \| SkillTemplateFn` | The per-spawn agent preamble — *how* an agent should investigate/act with this AgentApp. Rendered via Eta with the agent-render context. |
| **examples?** / **hints?** | optional | `examples` is discipline/few-shot rendered per-spawn (never enters the shared spine). `hints` are UX/marketplace metadata. |

Notice what the anatomy implies: **an AgentApp ships with its placement in the model's memory hierarchy.** Its protocol is encoded once onto the shared spine, where every agent inherits it at zero re-encoding cost; its reference content stays corpus-resident, retrieved into live attention only when a task needs it; its tools cost nothing until invoked. Orchestration frameworks can't express this distinction — without a memory hierarchy, an integration's knowledge gets exactly one residency: re-sent with every call.

Both reference AgentApps follow this exact shape — **`corpus_research`** (local docs) and **`web_research`** (Tavily / keyless web search) — and [Your first AgentApp](/build-an-app/your-first-app) walks `createCorpusApp` line by line. Every concept on this page is real code you can read.

## The AgentApp protocol — what the model actually sees

Every enabled AgentApp contributes to the runtime's **shared spine** — the KV prefix decoded once and inherited by every forked agent. Five constants in `@lloyal-labs/rig`'s `protocol.ts` are the entire bytes-locked surface the model encounters:

| constant | role |
| --- | --- |
| `FRAMEWORK_INTRO` | The intro paragraph at the top of every spine. |
| `CATALOG_ENTRY(name, tools[], useWhen)` | Per-AgentApp catalog block emitted under `# Protocols`. |
| `TOOL_SELECTION_RULE` | The model's routing rule across enabled AgentApps. |
| `BOUNDARY_MARKER(name)` | `Apply the **<name>** protocol.\n\n` — prefixes every per-spawn message. |
| `APP_PROTOCOL_VERSION` | Bumped only when the marker/catalog/rule changes. |

Every byte of framework-rendered prompt content the model sees traces to one of those constants (or to grammar-sanitized `app.json` metadata interpolated through them). [Continuous Context](/under-the-hood/continuous-context-spine) walks the spine's physical layout.

## What `defineApp` enforces

The wiring step is also a security boundary. `defineApp` validates the spec synchronously, before the AgentApp enters the registry — so a malformed AgentApp can never ride the prompt:

- **Manifest schema.** `name`/`protocol.name` match `[a-z][a-z0-9_-]{1,63}`. `protocol.tools[]` is non-empty, unique, same regex.
- **`useWhen` sanitization.** Single bounded sentence (≤280 chars), no chat-role markers, no markdown code fences, no newlines. AgentApp-supplied strings can't break out of the catalog block.
- **AgentApp protocol version.** `manifest.appProtocolVersion` must be in `SUPPORTED_APP_PROTOCOL_VERSIONS` (currently `"3.0"`).
- **Tool-map coverage.** `Object.keys(tools)` equals `protocol.tools[]` as a set, and each instance's `.name` matches its key — the catalog's `Tools:` line and the dispatcher must agree.
- **Boundary-marker double-emission.** If `skill` is a string, it MUST NOT contain the literal `Apply the **` substring — the framework prepends the marker, so a `skill.eta` carrying the line would emit it twice.

These run synchronously and throw on failure. AgentApp factories should call `defineApp` **last** (after constructing source + tools) so a bad manifest fails at construction, not later at registration.

## Lifecycle — no hooks, just scopes

There are no `install`/`uninstall`/`enable`/`disable` hooks on the AgentApp. Lifecycle is Effection scope:

- **Factory body = setup.** An `AppFactory` is `() => Operation<App>`. Read config + reranker, construct, return.
- **`ensure(...)` = teardown.** Anything the factory allocates that needs cleanup (a connection pool, a watcher) goes inside `ensure(...)` so the registry's scope-halt fires it.
- **Per-AgentApp detached scope.** The registry runs each factory in its own detached scope and tears them down in reverse register-order on registry-scope exit — best-effort, so a throwing teardown can never strand a sibling or crash the harness.

[Lifecycle & registry](/build-an-app/lifecycle-and-registry) walks the registry's scope model in detail.

## Security at a glance

AgentApps run in-process with the model, so the trust boundary is **the prompt itself**:

- **`Tool.protected`** marks sensitive (write) tools. The framework's `authGuard` denies a protected tool unless the session holds a **grant** for it (`GrantStoreCtx`). The credential never enters the model's context; the model only triggers the call.
- **Metadata sanitization** at `defineApp` time prevents an AgentApp's strings from breaking out of the catalog block (the regex/forbidden-pattern checks above).
- **Per-spawn prose isolation.** Each agent's per-spawn user message is prefixed with `Apply the **<name>** protocol.` — the model knows which AgentApp's discipline applies to *this* turn, and no other AgentApp's `skill`/`examples` enters the rendering.

[AgentApp security](/build-an-app/app-security) covers the threat model and the full defense stack.

## Distribution

An AgentApp is a **signed npm tarball**. First-party AgentApps that ship inside the HDK reference harness (`reasoning.run`) are ordinary build-time deps — the harness `import { createWebApp } from '@lloyal-labs/web-app'` and enables the factory with `registry.enable`. Third-party AgentApps — and any AgentApp reaching a harness it doesn't own — flow through the canonical **signed-tarball channel** at `apps.lloyal.ai`: `harness.dev install <publisher>/<app>` signature-verifies and `npm install`s the tarball into `node_modules`, and the harness imports its factory the same way. The catalog URL and the trust roots used to verify signatures are **compile-time constants in `@lloyal-labs/rig`** — the harness does not configure them.

The runtime primitives live in `@lloyal-labs/rig` (`resolveAppEntry`, `verifyBundle`, `CHANNEL_CATALOG_URL`, `CHANNEL_TRUST_ROOTS`) and back the install CLI's verify path; there is no runtime "load app by name" verb. The `harness.dev` CLI wraps them for scaffolding, publishing, and installing AgentApps. [Distribution](/build-an-app/distribution) covers the full flow.
