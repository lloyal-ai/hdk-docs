A **Source** is the data-access half of an AgentApp. It wraps a live system (a corpus, the web, a vector store, a database) and exposes the tools an agent uses to read it ‚Äî plus the reranker-backed scorer that decides whether what came back is relevant.

This page is the deep dive on the `Source` abstract base, the four scoring roles of the cross-encoder reranker, and how Retrieval Interleaved Generation (**RIG**) puts retrieval *inside* the model's decode loop rather than before it.

## The `Source` base

```ts
export abstract class Source<TCtx = unknown, TChunk = unknown> {
  /** Human-readable source name. */
  abstract readonly name: string;

  /** Data-access tools the App exposes. */
  abstract get tools(): readonly Tool[];

  /** Reranker injected at construction; powers `createScorer`. */
  protected _reranker: ScorerReranker | null = null;

  /** Minimum logit-diff to proceed. Default `0` (model leans yes). */
  protected _entailmentFloor: number = 0;

  /** Create an immutable EntailmentScorer scoped to one invocation. */
  createScorer(originalQuery: string): EntailmentScorer { /* ... */ }

  /** Optional content advert read by the per-spawn skill template. */
  promptData(): unknown { return {}; }

  /** Optional reranker-bound chunk getter for post-use scoring. */
  getChunks(): TChunk[] { return []; }
}
```

Two reference subclasses ship with the HDK:

- **`CorpusSource`** (`@lloyal-labs/corpus-app`) ‚Äî owns the loaded resources + chunked tokens + the reranker. Exposes `grep`, `read_file`, `search`. `promptData()` returns the corpus TOC.
- **`WebSource`** (`@lloyal-labs/web-app`) ‚Äî owns a `SearchProvider` (Tavily or keyless DDG) + optional reranker. Exposes `web_search`, `fetch_page`.

A custom Source for a vector store, a database, a JIRA project ‚Äî same shape, same `defineApp` protocol.

## Why `promptData()` matters

The skill template (`it.toc`, `it.title`, ‚Ä¶) reads from `promptData()`. That's how the AgentApp tells the agent *what kind of source this is* before the agent searches. Without it, the agent has to discover the source's identity by probing ‚Äî and probes can mislead.

For `corpus_research` the TOC carries the file list + top-level headings; the recon agents read that as "the corpus *is* the HDK docs" rather than guessing from chunk content. AgentApps without a static content advert (the web) can return `{}` and the template renders a fallback.

[Why this matters in practice ‚Äî preflight recon](/build-a-harness/reference-architecture) walks the failure mode without `promptData`.

## The reranker ‚Äî four scoring roles

The HDK ships a single cross-encoder reranker context (default: **Qwen3-Reranker-0.6B**) that drives an `EntailmentScorer` with four distinct methods. One model, four roles ‚Äî the reranker is RIG infrastructure, not a fetch optimization.

```ts
export interface EntailmentScorer {
  /** Score texts against the original query. */
  scoreEntailmentBatch(texts: string[]): Promise<number[]>;

  /** Dual: min(local query score, original query score). Exploit mode. */
  scoreRelevanceBatch(texts: string[], localQuery: string): Promise<number[]>;

  /** Score texts against an arbitrary reference. Echo detection at delegation. */
  scoreSimilarityBatch(reference: string, texts: string[]): Promise<number[]>;

  /** Floor gate ‚Äî true if the score clears `_entailmentFloor`. */
  shouldProceed(score: number): boolean;
}
```

### Three queries in play

| Concept | Scope | Field name in code |
|---|---|---|
| **Tool query** | Per tool call | `localQuery` param of `scoreRelevanceBatch` |
| **Agent task** | Per agent lifetime | `reference` param of `scoreSimilarityBatch` |
| **Original query** | Per research invocation | Captured in closure by `createScorer` |

Conflating these produces wrong scores. The four methods correspond to four boundaries:

- **`scoreEntailmentBatch`** ‚Äî does this retrieved content stay on the *original* research intent? Used at content-prefill boundaries.
- **`scoreRelevanceBatch`** ‚Äî combine the agent's per-tool query with the original query via `min(local, original)`. Used in exploit mode when KV pressure tightens focus.
- **`scoreSimilarityBatch`** ‚Äî compare texts against an arbitrary reference (the agent's own task, typically). Powers echo detection at delegation boundaries ‚Äî preventing an agent from re-issuing a search that paraphrases its own task.
- **`shouldProceed`** ‚Äî the floor gate. Below it, the content is dropped or the agent stalls.

## The score formula ‚Äî logit-diff, floor `0`

The reranker's `score(query, doc)` returns a **logit-diff**: `logit(yes) ‚àí logit(no)`. Range unbounded. Sign carries meaning:

- **Positive** ‚áí the model would emit "yes" rather than "no" if forced ‚Äî relevant.
- **Negative** ‚áí "no" ‚Äî irrelevant.
- **Magnitude** ‚áí confidence.

The default `_entailmentFloor` on `Source` is `0`: a hit passes the gate when the cross-encoder leans toward "yes." Subclasses can tighten (`2` for "confident yes" in noise-heavy corpora) or loosen (negative floors for sparse data where leaning-no hits are still worth investigating).

<Note>
The prior softmax form returned `P(yes | yes_or_no) ‚àà (0,1)` with a `0.25` default floor ‚Äî small logit gaps got compressed to extreme probabilities (a gap of 5 ‚Üí 0.993; 10 ‚Üí 0.99995) which saturated top-K ordering and erased confidence differences in dense result sets. Logit-diff preserves the dynamic range. See the calibration probe at `reasoning.run/scripts/inspect-rerank.mjs` and [Rerank architecture](/under-the-hood/rerank-architecture) for the empirical evidence.
</Note>

## Where scoring happens

Scoring fires at three points in a pool's lifecycle, all inside the dispatch fiber:

1. **Search-time scoring (inside the tool).** `SearchTool.execute` calls `reranker.score(query, chunks)`. Hits below the threshold are dropped; the result envelope `{ hits, thresholdScore, totalScored, topRejected }` lets the agent see "best I could find, but the model said no to all of them" instead of getting fed garbage as honest signal.
2. **Boundary entailment at SETTLE.** When a tool result is prefilled back into the agent's KV, the framework optionally re-scores it against the *original query* via `scoreEntailmentBatch` ‚Äî a content boundary check that catches off-topic drift after the per-tool scoring already passed.
3. **Echo detection at delegation.** When an agent forks a sub-agent, the parent's task is the `reference` and the candidate sub-task is compared via `scoreSimilarityBatch` ‚Äî preventing the recursion from spinning on paraphrases of the same intent.

The single reranker context handles all three concurrently via its `_drain` + `_fillGroup` multiplexing ‚Äî multiple requests in the pending queue get packed into shared GPU batches of `nSeqMax` slots.

## RIG: retrieval inside the loop

Classical RAG is a *pre-processing* step:

```
embed(query) ‚Üí vector search ‚Üí concat(top_K, prompt) ‚Üí generate
```

One shot. Static context. The model never re-shapes retrieval based on what it learned.

RIG inverts this ‚Äî retrieval happens **inside** the generation loop:

<Steps>
  <Step title="Agent generates a tool call">
    PRODUCE phase samples a `search` (or `web_search`, `grep`, ‚Ä¶) call.
  </Step>
  <Step title="Source returns chunks">
    The AgentApp's `Source` queries the backend.
  </Step>
  <Step title="Reranker scores">
    Cross-encoder scores chunks against the original query as a logit-diff. Hits below the floor are dropped.
  </Step>
  <Step title="Prefill at the call position">
    SETTLE phase prefills the result into the agent's KV at the position where the call was made.
  </Step>
  <Step title="Attend and refine">
    Next PRODUCE attends over the prefilled result. The next query is shaped by what the model *just read*.
  </Step>
</Steps>

The dynamic isn't a prompt trick ‚Äî it's a consequence of the attention mechanics. The model's Q¬∑K·µÄ for its next token attends over all K vectors at positions 0..N including those written during the tool-result prefill. Later searches reference concepts **absent from the original question** because they emerged from earlier reads still physically present in the KV cache.

### Pressure-aware exploit/explore

The framework's `DefaultAgentPolicy` exposes an explore/exploit gate that flips reranker behaviour when KV runs short:

- **Explore mode** (default, headroom available). `SearchTool` ranks chunks against the *tool query* alone ‚Äî agents discover bridging content adjacent to the answer even when the original query wouldn't surface it.
- **Exploit mode** (under pressure). `SearchTool` re-scores via `scoreRelevanceBatch` ‚Äî `min(toolQueryScore, originalQueryScore)` ‚Äî tightening focus at the cost of serendipitous discovery.

The agent doesn't know which mode it's in; the policy decides and the tool obeys. [KV pressure](/under-the-hood/kv-pressure) walks the budget model.

## Implementation tips

When authoring a custom Source:

- **Bind the reranker at construction.** Set `this._reranker` in the constructor so `createScorer` and the search tool both see it. The corpus and web AgentApp factories show the pattern.
- **Heavy setup in the AgentApp factory, not in `tools`.** Tokenizing chunks for the reranker is expensive. Do it once when the factory builds the source, not per-search.
- **`promptData()` is the protocol surface.** Whatever this returns is what the AgentApp's skill template can render. Treat the keys as your prompt API.
- **Honor the floor.** When you write a custom `SearchTool`, return only hits above `this._entailmentFloor` in the `hits` array, and surface dropped hits in `topRejected` so the agent has an honest "no good matches" signal.
