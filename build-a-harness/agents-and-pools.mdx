**One `agentPool` call runs N agents in the same GPU batch.**

```typescript
import { agentPool, parallel } from "@lloyal-labs/lloyal-agents";

const pool = yield* agentPool({
  orchestrate: parallel(
    questions.map((q) => ({ content: q, systemPrompt: RESEARCH_PROMPT })),
  ),
  tools: [searchTool, fetchTool, reportTool],
  terminal: reportTool,
});
// pool.agents[0].result, pool.agents[1].result, ...
```

```
Spine (system prompt + tool schemas): decoded once
  ├── Agent 0: fork → investigate question A
  ├── Agent 1: fork → investigate question B
  └── Agent 2: fork → investigate question C
```

Every agent forks from the spine's KV cache — an O(1) metadata tag, not a copy. All active agents advance together, one batched decode per tick. Swap `parallel` for `chain`, `fanout`, or `dag` (or any custom `Orchestrator` generator) and the same call changes topology. The rest of this page is the knobs, in the order you'll reach for them.

This page documents `create-agent-pool.ts`, `agent-pool.ts`, and `orchestrators.ts` in `@lloyal-labs/lloyal-agents`.

## What is an Agent?

An agent is a **branch with intent**. A Branch is a forkable KV cache sequence — it stores every token the model has seen and generated. An Agent adds: a task, a policy for accomplishing it, and a record of what it has done. The branch is the ground truth; the agent is the interpretation layer.

```typescript
// agent and agentPool return Agents. After completion, each exposes:
result.result          // what the agent discovered (or null)
result.rawOutput       // raw model output from the final turn
result.toolCallCount   // how many tools it called
result.tokenCount      // how many tokens it generated
result.nestedResults   // results bubbled up from recursive sub-agents
result.ppl             // model-level perplexity at completion
```

Three pieces of agent state matter for everything downstream:

- **Status**: `idle → active → awaiting_tool → active → … → idle → disposed`. The pool enforces valid transitions; a final `idle` with a live branch is the recovery-extraction window.
- **Result provenance**: every result records how it was produced — `voluntary_return` (terminal tool), `free_text`, `recovery` (extracted after a kill), or `tool_error`.
- **Tool history + lineage**: each tool invocation is recorded (name, args, result token count, context-after). When a tool spawns sub-agents, each sub-agent knows its caller; `walkAncestors()` traverses self → caller → caller's caller, and tool guards use that chain for cross-level deduplication.

## Single agent: `agent`

One agent, one task, automatic lifecycle:

```typescript
import { agent } from "@lloyal-labs/lloyal-agents";

const result = yield* agent({
  systemPrompt: "You are a research assistant.",
  task: "Find information about voice agent architectures.",
  tools: [searchTool, fetchTool],
  terminal: reportTool,
});

console.log(result.result);
```

### Structured output with `schema`

For single agents that should produce JSON instead of calling tools:

```typescript
const result = yield* agent({
  systemPrompt: "You analyze research queries. Output JSON only.",
  task: `Analyze: "${query}"`,
  schema: {
    type: "object",
    properties: {
      questions: { type: "array", items: { type: "string" } },
    },
    required: ["questions"],
  },
});
const plan = JSON.parse(result.rawOutput);
```

The schema compiles to a GBNF grammar applied **eagerly** on the generating branch — every token is grammar-valid from the first sample. No retry logic, no post-hoc validation. (`PlanTool` in `@lloyal-labs/rig` is this pattern productized.)

### Warm path via `session`

When a `Session` has a trunk (previous turns committed), pass it to continue from prior conversation state:

```typescript
const result = yield* agent({
  systemPrompt: EVAL_PROMPT,
  task: evalQuestion,
  schema: evalSchema,
  session, // forks from session.trunk — prior Q&A is in KV
});
```

## `agentPool` options, by significance

### Required

- **`orchestrate`** — the execution shape: `parallel(specs)`, `chain(items, toStep)`, `fanout(landscape, domains)`, `dag(nodes)`, or a custom generator over `PoolContext` (`spawn` / `waitFor` / `extendSpine` / `canFit`).
- **`tools`** — `Tool[]`, the data-access instances. Optional in the type, practically required.
- **`terminal`** — the tool that ends an agent's turn, **by reference** (e.g. `reportTool`). The framework merges it into the tool set so its schema reaches the model, then intercepts its call at the policy layer and captures the `result` argument as the agent's return — the terminal's `execute()` never runs. Omit it and agents end on free-text/stop, which is heavier and less clean.

### "What about follow-up questions?" — `session`

`Session` (from `@lloyal-labs/sdk`) is **the** multi-turn knob. It owns the **trunk** — a long-lived KV branch. After a pool exits, call `session.commitTurn(query, answer)`; the next pool forks from the extended trunk and reads prior Q&A through attention, not through re-injected prompt history.

```typescript
import { Session } from "@lloyal-labs/sdk";

const session = new Session({ ctx, store });

yield* agentPool({ orchestrate, tools, session });
yield* call(() => session.commitTurn(query1, answer1));

// Follow-up — agents fork from session.trunk, see prior Q&A
yield* agentPool({ orchestrate: parallel([followup]), tools, session });
```

The trunk is the only branch that survives across queries; everything else is pruned on scope exit. [Sessions](/build-a-harness/sessions) has the full API.

### "What about sub-agents?" — `parent`

`agentPool({ parent: someBranch })` forks the pool's spine from that branch instead of cold-starting at position 0. Warm-path precedence: **`parent` > `session.trunk` > cold start**. `DelegateTool` uses this internally with the calling agent's branch; you'd reach for it directly when threading spines between harness stages.

### "How do I tune budgets and recovery?" — `policy`

`DefaultAgentPolicy` with no options is fine to start. Customize when you need budgets, recovery, or extra guards:

```typescript
const policy = new DefaultAgentPolicy({
  budget: {
    context: { softLimit: 2048, hardLimit: 1024 }, // KV-tokens remaining
    time: { softLimit: 240_000, hardLimit: 360_000 }, // ms — NO time budget unless you set one
  },
  recovery: { prompt: RECOVERY_PROMPT }, // extract findings from killed agents
});
```

The full decision surface — enforcement points, guards, nudges, recovery, retry parking — lives in [Pressure & policy](/under-the-hood/kv-pressure). The two facts you need here: KV defaults are softLimit 1024 / hardLimit 512 (and `hardLimit` must be ≥ the context's `nBatch` — the pool throws at startup otherwise), and there is **no time budget by default**.

### "How long do agents run?" — `maxTurns`

Default 100 — almost always too generous; research pools cap at ~10. Per-agent, not pool-wide. A turn is one tool-call → result-settle round.

### "KV is filling up mid-pool" — `pruneOnReturn`

Set `true` for parallel pools: an agent's branch is pruned the moment it voluntarily reports, freeing its unique KV cells for the siblings still working. Hard-cut agents keep their branches for recovery extraction.

### "Many agents, same role and tool set" — `systemPrompt` (shared mode)

When set, the chat header (system + tool schemas) is prefilled onto the pool's inner spine **once**; every agent inherits it via metadata-only prefix-share and its own suffix carries only the task. Use for single-role pools (chain-mode research, same-role fan-outs); skip for mixed-role workflows.

<Warning>
**Shared mode changes where `extendSpine` writes.** In shared mode, extensions land on the pool's inner spine — visible to later agents in the pool, but **discarded when the pool closes**. In non-shared warm mode, extensions write onto the `parent` you passed — they persist, and a later synth pool forking the same parent attends over them. If your pipeline is research-pool → synth-over-findings, put the shared catalog on the harness's outer `withSpine` and let the pool run non-shared against it.
</Warning>

See [Mixed-role pools](/under-the-hood/mixed-role-pools) for the full convention.

### "I'm running a thinking-mode model" — `enableThinking`

Default `false`. Set `true` for Qwen3-family thinking models: the chat template emits `<think>` delimiters and the parser separates reasoning from content. The flag is captured per-agent and threaded through every tool-result prefill — mismatched values corrupt the parse.

### RIG knobs

- **`scorer`** — an `EntailmentScorer` from `Source.createScorer(originalQuery)`. The pool never reads it; it rides `ToolContext.scorer` into every dispatch at every recursion depth. Tools that admit retrieved text into KV (search, fetch, delegation) consult it; everything else ignores it. Canonical reference: [Sources & retrieval](/build-an-app/sources-and-retrieval).
- **`echoThreshold`** (default 0.8) / **`checkAncestorEcho`** (default false) — the delegation echo gate; see Deep agents below.

### Diagnostic

- **`trace`** — adds per-token entropy and surprisal to `agent:produce` events. The trace writer itself is set once via `initAgents({ traceWriter })`.

## Deep agents

A sub-agent here is not a fresh conversation handed a brief — it's a fork of live attention. Recursion is structural, at two levels:

**Model level — tools that spawn pools.** `Tool.execute()` returns an Effection `Operation`, so a tool body can `yield*` straight into `agentPool`. When the model calls such a tool, sub-agents spawn mid-generation, inside the calling agent's scope, forked from the calling agent's branch (`context.branch`) — inheriting every search result, page read, and reasoning step for the cost of a turn separator. `DelegateTool` from `@lloyal-labs/rig` is the canonical implementation:

```typescript
import { DelegateTool } from "@lloyal-labs/rig";

const delegate = new DelegateTool({
  name: "web_research",
  description: "Spawn parallel sub-agents to investigate sub-questions.",
  extractTasks: (args) => args.tasks as string[],
  systemPrompt: RESEARCH_PROMPT,
  poolOpts: { tools: [searchTool, fetchTool], terminal: reportTool },
});

const pool = yield* agentPool({
  orchestrate: parallel(specs),
  tools: [searchTool, fetchTool, delegate],
  terminal: reportTool,
  scorer, // propagates to every depth via ToolContext
});
```

**Harness level — pools whose branches outlive them.** `useAgentPool` is a resource: after the agents complete, their branches stay alive until your scope exits, so a follow-up pool can fork from any completed agent's frontier.

**The delegation boundary is judged, not just crossed.** Because sub-tasks are proposed against live attention, `DelegateTool` scores them before spawning anything: entailment gating drops sub-tasks that drifted from the original query, and echo detection (`scoreSimilarityBatch` against the calling agent's own task, threshold `echoThreshold`) rejects paraphrase recursion outright — the agent is told to report what it found instead of re-delegating. With `checkAncestorEcho`, the check walks the whole lineage.

**Results bubble up.** Any tool returning `{ results: string[] }` or `{ nestedResults: string[] }` feeds the calling agent's `nestedResults` — `pool.agents[i].nestedResults` collects what every depth found, no tool-name hardcoding.

Depth is unbounded by design; KV pressure is the budget. Every subtree is a structured-concurrency scope — halt it and its branches are reclaimed.

## Recovery

Agents killed before reporting (pressure, turn cap, time budget) don't lose their work. Configure `recovery` on the policy and the pool prefills an extraction nudge into the agent's **own branch** — the full investigation is still in KV — under an eager `{ result: string }` grammar:

```typescript
const policy = new DefaultAgentPolicy({
  recovery: {
    prompt: { system: "Call report with your findings.", user: "Report. Budget: <%= it.budget %> words." },
    minTokens: 100,   // skip agents with less than this generated (default 100)
    minToolCalls: 2,  // skip agents with fewer tool calls (default 2)
  },
});
```

Recovered results carry `resultSource: 'recovery'` and emit `agent:recovered` (not `agent:return`). Mechanics, budget formula, and trace events: [Pressure & policy](/under-the-hood/kv-pressure).

## `useAgent` and `useAgentPool`

`agent` and `agentPool` are the scoped wrappers — branches are pruned when they return. When branches must stay alive after completion (verification forks, follow-up pools), use the resources:

```typescript
import { useAgent, useAgentPool, createToolkit } from "@lloyal-labs/lloyal-agents";

// useAgent — resource; agent.branch stays alive until your scope exits
const agent = yield* useAgent({ systemPrompt, task, tools: [searchTool] });
const verifyBranch = agent.branch.forkSync();
```

`useAgentPool` is the low-level primitive `agentPool` wraps. It takes the pre-compiled shapes (a spine `Branch`, a `Map` of tools, the schemas JSON, a terminal **name**) and returns a `Subscription<AgentEvent, AgentPoolResult>` — events stream during execution, the result is the close value:

```typescript
const toolkit = createToolkit(tools, reportTool);
const sub = yield* useAgentPool({
  spine,
  orchestrate: parallel(specs),
  tools: toolkit.toolMap,
  toolsJson: toolkit.toolsJson,
  terminalToolName: toolkit.terminalName,
});
let next = yield* sub.next();
while (!next.done) {
  handleEvent(next.value); // AgentEvent
  next = yield* sub.next();
}
const pool = next.value; // AgentPoolResult
```

Most code never touches these — `agent` and `agentPool` handle the toolkit, the Subscription drain, and scope cleanup.

## Agent Policy

The pool consults an **AgentPolicy** at every decision boundary. The interface, from `AgentPolicy.ts`:

```typescript
interface AgentPolicy {
  onProduced(agent, parsed, pressure, config): ProduceAction;   // stop token: return / nudge / tool_call / idle
  onSettleReject(agent, resultTokens, pressure, config): SettleAction; // oversized result, no siblings to free KV
  shouldExit?(agent, pressure): boolean;     // pre-produce kill gate
  shouldExplore?(agent, pressure): boolean;  // explore vs exploit per dispatch
  onRecovery?(agent, pressure): RecoveryAction;
  onToolRetry?(agent, tool, error, attempt): ToolRetryAction;   // park-and-retry transient failures
  resetTick?(): void;
  readonly pressureThresholds?: PressureThresholds;
}
```

`DefaultAgentPolicy` ships the standard behavior: terminal-tool interception, budget nudges ("report your findings now within N words"), confabulation prevention (a return before `minToolCallsBeforeReturn` non-terminal calls is nudged back to work), declarative tool guards, explore/exploit switching, and transient-failure parking (a tool that throws `ToolRetryError` — a rate-limited provider — parks the agent at zero KV cost and re-executes the same call after the delay; the model never sees infrastructure weather).

```typescript
const policy = new DefaultAgentPolicy({
  minToolCallsBeforeReturn: 3,
  extraGuards: [myCustomGuard],
  shouldExplore: { context: 0.4 }, // exploit below 40% KV available
  budget: { context: { softLimit: 2048 }, time: { softLimit: 600_000 } },
  recovery: { prompt: RECOVERY_PROMPT },
  maxToolRetries: 1,
});
```

### Tool Guards

Guards are declarative rejection rules checked **before** budget nudges and before dispatch — a stuck agent gets the specific message ("this URL was already fetched"), not a generic turn-limit nudge:

```typescript
interface ToolGuard {
  tools: string[] | '*';   // '*' sees every call (the framework's authGuard uses this)
  reject: (args, lineageHistory, agent, toolName, config) => boolean;
  message: string;
}
```

Defaults: the framework-injected **authGuard** (denies `protected` tools without a session grant — see [App security](/build-an-app/app-security)), `fetch_page` URL dedup, and `web_search` query dedup — both checked against the agent's full lineage history. Sibling-level dedup is the tools' job via `ToolContext.peerHistory` ("Resource unavailable. Try a different query.").

Drift and echo scoring at delegation boundaries is the scorer's job — canonical treatment in [Sources & retrieval](/build-an-app/sources-and-retrieval).

## Context pressure

Pools share one KV cache; `ContextPressure` snapshots it every tick, and two thresholds (softLimit 1024 / hardLimit 512 by default) govern nudges, settles, and kills. Pressure management is automatic — tools never check headroom. The full budget model, enforcement points, and the symptom→fix table live in [Pressure & policy](/under-the-hood/kv-pressure).

## Prefix sharing

Forking is metadata-only; what the fork *inherits* depends on mode. Default: the spine carries only a turn separator and each agent formats its own `[system + tools + user]` suffix. Shared mode: the header is decoded once on the spine and suffixes shrink to the task.

```
Default:       3 agents × ~900 tokens of prompt+tools ≈ 2,700 unique cells
Shared mode:   ~900 spine + 3 × ~30 per-agent suffix ≈ 990 unique cells
```

When `DelegateTool` spawns sub-agents from a parent that has already read 760 tokens of pages, the sub-agents prefill a 1–2-token turn separator — not 760 tokens. Full physics: [Continuous Context](/under-the-hood/continuous-context-spine).

## Events

The pool streams events through the `Events` channel:

| Event | Payload highlights |
| --- | --- |
| `agent:spawn` | `agentId`, `parentAgentId` |
| `agent:produce` | `text`, `tokenCount`, optional `entropy`/`surprisal` |
| `agent:tool_call` / `agent:tool_result` | tool, args / result, `contextAvailablePercent` |
| `agent:tool_progress` | progressive feedback (e.g. reranker scoring) |
| `agent:tool_retry` | transient failure parked; `retryAfterMs`, `attempt` |
| `agent:return` | voluntary completion via terminal tool |
| `agent:recovered` | findings extracted from a killed agent |
| `agent:done` | agent finished (any path) |
| `agent:tick` | batch decode committed; `cellsUsed`, `nCtx` |

```typescript
yield* spawn(function* () {
  for (const ev of yield* each(events)) {
    if (ev.type === "agent:return") console.log(`Agent ${ev.agentId}: ${ev.result}`);
    yield* each.next();
  }
});
```
