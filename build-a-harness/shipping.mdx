---
title: Ship your harness
description: Getting a harness to end users — as a CLI on npm, inside an Electron app, or as a worker in an existing backend — and what to do about the model weights.
---

A harness is a Node program with a native dependency, so it ships the way Node programs ship. Three proven paths, in rising order of effort — plus the one question all of them share: *where do the weights come from?*

## The invariants (all three paths)

- **Never bundle the native binding.** `@lloyal-labs/lloyal.node` ships prebuilt binaries for thirteen platform/GPU targets and must stay a real npm dependency so the right binary resolves at install time. If you bundle your harness with esbuild, pass `--packages=external`.
- **Never ship weights in your package.** Model files are gigabytes; npm and app stores are the wrong transport. Either ask for a path (the scaffold's `harness.json` pattern) or download on first run (below).
- **AgentApps reproduce from the lockfile.** Signed-channel apps are recorded in `package-lock.json` with the canonical URL + sha512, so any standard pipeline — Docker, CI, `npm ci` — rebuilds your dependency tree with [no `harness.dev` in the deploy path](/cli/installing).
- **One harness process, one user's work.** The runtime is single-tenant by design — determinism and multi-tenant batching don't mix. A harness serves *sessions and outcomes*; if you need to serve many users, run many processes, not one shared context.

## Path 1 — a CLI on npm

The scaffold is already wired for this: `bin/run.js` requiring the compiled output, a `build` script, and a `files` whitelist. From there to `npx your-harness` is a `package.json` away:

```json
{
  "name": "your-harness",
  "bin": { "your-harness": "bin/run.js" },
  "files": ["bin/", "dist/", "prompts/", "LICENSE", "README.md"],
  "scripts": { "build": "tsc", "prepublishOnly": "npm run build" }
}
```

For a single-file distribution, bundle with esbuild instead of `tsc` — keeping packages external so `lloyal.node` resolves natively:

```bash
esbuild src/main.ts --bundle --platform=node --format=esm \
  --packages=external --outfile=dist/bundle.mjs
```

This is exactly how the reference research harness ships (`npx reasoning.run`): esbuild bundle, bin redirect, prompts as files, framework packages as real dependencies.

## Path 2 — inside a desktop app

**Artifact is the existence proof** — a signed, notarized Electron app on [apps.lloyal.ai](https://apps.lloyal.ai) built on the same packages you're using. The architecture that works:

- **Run the engine out of the renderer.** The harness (model, agents, pools) lives in a `utilityProcess` (or equivalent worker); the UI talks to it over an IPC event bus. The pool's event stream maps 1:1 onto IPC messages — the renderer is a reducer over `agent:*` events, exactly like a TUI would be.
- **Package with electron-builder** (or Forge). The native binding rides along as a normal dependency; the platform prebuilds mean no user-machine compilation.
- **Sign and notarize.** A local-AI app asks for real trust; an unsigned binary undercuts the whole privacy story. Budget for the platform's signing ceremony (Developer ID + notarization on macOS).
- **Download weights on first run** with a visible progress state, and verify what you fetched. Check available memory before allocating the context so a too-small machine gets a clear message instead of a hang.

## Path 3 — a worker in an existing backend

The harness is an npm dependency like any other — `import` it inside the service you already deploy. The deltas from a laptop:

- **Provision for the working set**: weights + KV cache + reranker resident per process ([Deployment](/guides/deployment) has the sizing tables).
- **One process per concurrent workload** (the single-tenant invariant). A job queue feeding N harness processes scales horizontally; one process multiplexing N users does not.
- **Nothing else changes.** `npm ci` from the committed lockfile reproduces everything, including signed-channel AgentApps. Your agent turns still never leave the box — tasks in, results out is your service's API, not the model's.

## Weights: the first-run contract

Whichever path, decide the model-acquisition UX up front:

| Strategy | Fits | Notes |
|---|---|---|
| **User supplies a path** | CLIs, developer tools | The scaffold's default (`harness.json` / `--model`). Zero download logic; assumes a user who has GGUFs. |
| **Download on first run** | Desktop apps, appliances | Fetch from a URL you control or Hugging Face at first launch; show progress; cache in a well-known directory; verify size/hash before loading. |
| **Preloaded environment** | Air-gapped, fleet deploys | Weights arrive with the machine image or on media; the harness only ever reads a local path. Pairs with the [zero-egress posture](/start-here/privacy-and-zero-egress). |
