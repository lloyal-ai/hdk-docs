**Every prompt, tool call, agent turn, and pressure transition тАФ captured as JSONL you can grep, pipe through Python, and reconstruct into the full execution tree.**

```typescript
import { initAgents, JsonlTraceWriter } from "@lloyal-labs/lloyal-agents";
import * as fs from "node:fs";

const traceWriter = new JsonlTraceWriter(
  fs.openSync(`trace-${Date.now()}.jsonl`, "w"),
);
const { session, events } = yield* initAgents(ctx, { traceWriter });
```

When no trace writer is passed, `initAgents` uses a `NullTraceWriter` тАФ every trace call is a no-op, zero cost. (The reference harness wires this for you: `npx reasoning.run --output-dir ./runs` writes one trace per session.)

## What gets captured

Every event has a `traceId`, `parentTraceId` (for building the tree), and `ts`. The full union lives in `packages/agents/src/trace-types.ts`; the ones you'll grep first:

| Event | What it captures |
|-------|-----------------|
| `scope:open/close` | Named scope boundaries with duration тАФ pools, tools, spines |
| `prompt:format` | The exact prompt the model sees + the raw task content before formatting |
| `agent:turn` | Raw model output + parsed content + parsed tool calls per turn |
| `tool:dispatch` | Tool name, parsed args, **toolkit position** (`toolIndex/toolkitSize`), explore flag, pressure |
| `tool:result` / `tool:error` / `tool:retry` | Result + prefill token count + duration; errors; transient-failure parking |
| `tool:authReject` | Protected-tool denial тАФ attempted tool, AgentApp label, full lineage history |
| `pool:open/close` | Agent count, pressure snapshot, per-agent findings |
| `pool:agentDrop` | Why an agent was killed тАФ the [drop-reason enum](/under-the-hood/kv-pressure#drop-reasons) |
| `pool:recoveryProduce` тЖТ `pool:recoveryReturn` / `pool:recoveryFailed` | Recovery extraction diagnostics тАФ every attempt ends in exactly one of the pair |
| `spine:extend` | Orchestrator extensions тАФ user/assistant content, delta tokens, position after |
| `branch:create/prefill/prune` | KV lifecycle with token counts and roles |
| `bm25:start/end`, `rerank:start/end` | Retrieval funnel тАФ candidates in, kept, top scores, duration |
| `entailment:search` / `entailment:delegate` / `entailment:delegate:echo` / `entailment:content:exploit` | Scoring decisions at search, delegation, and content boundaries |
| `generate:start/end` | Single-branch generation (plan, eval) with output |

## The trace tree

Events form a tree via `parentTraceId`. The `TraceParent` Effection context automatically connects inner pools to the outer tool dispatch that spawned them тАФ no manual wiring:

```
scope:open(pool, outer)
  тФЬтФАтФА tool:dispatch(web_research, A0)      тЖР toolIndex: 4/5
  тФВ   тФЬтФАтФА scope:open(withSpine)            тЖР inner pool
  тФВ   тФВ   тФФтФАтФА scope:open(pool, inner)
  тФВ   тФВ       тФЬтФАтФА tool:dispatch(grep)      тЖР toolIndex: 2/5
  тФВ   тФВ       тФФтФАтФА tool:result(grep)
  тФВ   тФФтФАтФА scope:close(withSpine)
  тФФтФАтФА tool:result(web_research, A0)
```

## Problem: agents not using tools

**Symptom:** agents generate prose, never call tools, end via free-text.

1. Check `prompt:format` events with `role: 'agentSuffix'` (or `role: 'spine'` in shared mode). The `tools` field must carry the JSON-serialized schemas тАФ missing/empty means the model never saw them.
2. Check `agent:turn` тАФ `parsedToolCalls: []` with content means the model produced text but no tool-call syntax.
3. Verify the model supports tool calling: templates without tool support make `formatChatSync` return `CHAT_FORMAT_CONTENT_ONLY` / `CHAT_FORMAT_GENERIC`, and agent setup throws ("Model does not support tool calling"). If that error is being caught and suppressed, agents run tool-blind.

**Common causes:** empty `tools: []` on the pool; a base (non-instruction-tuned) model; an instruction-tuned model without tool-call training.

## Problem: early termination (agents report too soon)

**Symptom:** agents call 0тАУ1 non-terminal tools, then `report()` with shallow findings.

1. Check `tool:dispatch` for the terminal tool's `toolIndex`. A terminal tool **last** in the toolkit array triggers recency bias тАФ see [tool ordering](/under-the-hood/grammar-and-ordering#tool-ordering-sensitivity).
2. Check `pool:open`'s `pressure.headroom`. Tight headroom at pool start means agents get budget-nudged ("report your findings nowтАж") after their first tool call.
3. Check `pool:agentDrop` for `pressure_softcut`.

**Common causes:** tool ordering; a system prompt that says "answer the question" instead of "investigate thoroughly using the tools"; earlier agents' tool results having consumed the budget (raise `nCtx`, lower agent count, or enable `pruneOnReturn`). Note the default policy already nudges first-turn reports back to work (`minToolCallsBeforeReturn: 2`).

## Problem: agents killed by pressure

**Symptom:** `pool:agentDrop` events; some agents produce no findings.

Group drops by `reason` and jump to the matching enforcement point in [Pressure & policy](/under-the-hood/kv-pressure) тАФ the symptomтЖТfix table at the top of that page is the canonical decoder. The trace-side signal chain: `pool:open` shows starting headroom; `tool:result.prefillTokenCount` shows the consumers (a `fetch_page` at 2,000+ tokens is typical); the drop reason names the phase.

**Solutions:** raise `nCtx`; fewer concurrent agents; `pruneOnReturn: true`; tools that compress results before returning (reranked chunk selection, not full pages); `q4_0` KV quantization to fit more tokens in the same memory ([Deployment](/guides/deployment)).

## Problem: recovery extraction fails

**Symptom:** killed agents have `result: null` despite real tool activity.

Every recovery attempt emits `pool:recoveryProduce` (token count + output length) followed by exactly one of `pool:recoveryReturn` or `pool:recoveryFailed`. On failure, the event carries `reason` тАФ `parse_error` / `no_result_field` / `scope_error` тАФ and a 200-char `outputExcerpt`, so you can diagnose without re-running:

```jsonl
{"type":"pool:recoveryProduce","agentId":3,"tokenCount":218,"outputLength":1204}
{"type":"pool:recoveryFailed","agentId":3,"reason":"parse_error: ...","outputExcerpt":"..."}
```

**No recovery events at all** for a killed agent means the policy skipped it: either `recovery` isn't configured on the policy, or the agent failed the confabulation gates (`minTokens: 100`, `minToolCalls: 2`). Drops via `maxTurns`/`pressure_softcut` recover only in the post-loop pass тАФ see the [decision matrix](/under-the-hood/kv-pressure#kill--nudge-decision-matrix) for which paths recover inline.

## Problem: synthesis ignores research findings

**Symptom:** synthesis output doesn't reference findings тАФ generic answers, or claims not present in the research.

1. Check the synth agent's `prompt:format` тАФ `taskContent` should carry the injected findings. Empty or "(none)" means the research phase produced nothing (check earlier `pool:close` events for non-null `result` per agent).
2. Check whether the synth agent has grounding tools: `tool:dispatch` events from a synth pool with `toolkitSize: 1` (report only) mean it can't verify anything itself тАФ it relies entirely on prompt content.

**Solutions:** verify research results landed before synth runs; give synthesis grounding tools (`tools: [...sourceTools, reportTool]`) when claims need verification; pass more reranked source passages into the synth prompt.

## Problem: the plan produces poor sub-questions

**Symptom:** research agents investigate tangential topics; sub-questions restate the query with different keywords or are too broad.

Find the planner's `generate:end` event and read the emitted JSON. Good sub-questions decompose into independently-investigable units. Fixes live in the plan prompt (be specific about decomposition criteria; add good-vs-bad examples) and in `PlanTool`'s `maxTasks` cap. An empty task list is the passthrough path тАФ correct for focused queries, wrong for multi-part ones.

## Debugging workflow

1. Run with tracing on; open the JSONL.
2. `pool:open` тАФ initial pressure and agent count.
3. `pool:agentDrop` тАФ premature kills; decode reasons via [Pressure & policy](/under-the-hood/kv-pressure).
4. `agent:turn` тАФ tool-call patterns: alternating search/read/grep (depth-first) or repeating one tool (scatter)?
5. `tool:result` тАФ unusually large `prefillTokenCount`?
6. `prompt:format` тАФ is the model seeing the right `taskContent` and `tools`?
7. `pool:close` тАФ how many agents produced results; total tokens.

To compare two runs, diff the tool-call sequences:

```bash
grep '"type":"tool:dispatch"' trace.jsonl | \
  jq -r '[.agentId, .tool, (.toolIndex|tostring)] | join("\t")'
```

Healthy pattern: `grep тЖТ search тЖТ read тЖТ grep тЖТ report`. Degenerate: `grep тЖТ report`.
